# 售后维护 SOP

适用对象：已上线的客户专属客服智能体。

维护目标：保证智能客服持续基于客户确认资料回答问题，及时补充知识库缺口，避免错误承诺和无人处理的转人工问题。

## 1. 维护角色

| 角色 | 责任 |
| --- | --- |
| 客户业务负责人 | 确认业务答案、价格规则、售后政策和禁止承诺内容 |
| 客户客服或销售 | 查看线索、跟进转人工、反馈高频问题 |
| 我方交付负责人 | 维护知识库、复测问题、记录版本变化 |
| 我方技术支持 | 处理入口不可用、发布异常、API 或嵌入问题 |

## 2. 日常维护节奏

| 频率 | 动作 | 负责人 |
| --- | --- | --- |
| 每天 | 查看是否有重要线索、投诉、紧急转人工 | 客户客服或销售 |
| 每周 | 汇总高频问题、回答错误、知识库缺口 | 客户业务负责人 / 我方交付负责人 |
| 每月 | 复盘对话质量、线索数量、转人工原因 | 双方负责人 |
| 业务变更时 | 更新价格、活动、服务范围、售后规则 | 客户业务负责人 |
| 入口异常时 | 检查 Dify 应用、嵌入页面、网络和发布链接 | 我方技术支持 |

## 3. 知识库更新流程

1. 客户收集新增问题或变更内容。
2. 客户给出确认后的标准答案。
3. 我方判断该内容是否适合由智能客服直接回答。
4. 对价格、合同、退款、投诉、效果承诺等内容增加边界。
5. 更新知识库 Markdown 或 Dify Knowledge。
6. 使用对应问题复测。
7. 记录更新时间、修改内容和复测结果。

## 4. 对话复盘流程

每周建议抽查以下对话：

- 用户问得最多的问题。
- 智能客服没有答上来的问题。
- 智能客服回答不准确的问题。
- 用户表达购买、预约、合作意向的对话。
- 触发投诉、退款、合同、报价的对话。
- 用户连续追问或明显不满意的对话。

复盘后按三类处理：

| 类型 | 处理方式 |
| --- | --- |
| 知识库缺口 | 补充标准答案并复测 |
| 业务规则未确认 | 交给客户确认，确认前转人工 |
| 不适合智能体回答 | 加入禁止承诺或转人工规则 |

## 5. 转人工处理 SOP

当智能体提示转人工或用户留下线索后，客户客服应检查：

- 用户称呼。
- 联系方式。
- 需求类型。
- 所在城市或服务区域。
- 用户最后一个未解决问题。
- 智能客服已给出的初步说明。

建议跟进话术：

> 你好，我看到你刚才咨询了我们的智能客服。这个问题需要人工进一步确认，我来帮你继续处理。

处理要求：

- 重要线索建议当天跟进。
- 投诉、退款、合同、正式报价类问题必须人工确认。
- 人工处理结果如形成稳定答案，应补充到知识库。

## 6. 常见问题处理

| 现象 | 可能原因 | 处理方式 |
| --- | --- | --- |
| 智能客服打不开 | Dify 发布链接、网络、嵌入页面异常 | 先测 Dify WebApp，再测客户页面 |
| 回答不准确 | 知识库缺少、冲突或检索不到 | 修正知识库，删除冲突内容，复测 |
| 回答太绝对 | 提示词或知识库缺少边界 | 改为范围、影响因素、人工确认 |
| 转人工太多 | 知识库覆盖不足或问题表达多样 | 补 FAQ，增加同义问法 |
| 留资太少 | 引导字段不清楚或时机不合适 | 调整线索收集话术 |
| 用户不满意 | 回答绕、慢、无具体下一步 | 补流程答案，明确人工入口 |

## 7. 不能直接更新的内容

以下内容必须客户业务负责人确认后才能加入知识库：

- 新价格、新活动、新优惠。
- 退款、赔偿、售后政策。
- 合同、发票、付款规则。
- 员工个人联系方式。
- 客户隐私数据或内部资料。
- 医疗、法律、金融等专业判断。
- 与平台规则或监管要求相关的承诺。

## 9. 升级触发条件

出现以下情况，可以进入二阶段增强评估：

- 客户希望聊天入口完全使用自定义 UI。
- 客户希望线索自动同步到 CRM、表格、飞书、企微。
- 客户希望不同门店、城市或业务线使用不同知识库。
- 客户希望统计转化率、来源渠道和客服质量。
- 客户需要更强的数据隔离、自托管或权限管理。

二阶段增强需重新确认范围、工期、费用、数据流向和安全边界。
