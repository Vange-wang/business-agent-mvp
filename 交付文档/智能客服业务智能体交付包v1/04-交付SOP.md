# 智能客服业务智能体交付 SOP

## 阶段 1：售前确认

目标：确认客户是否适合首版智能客服业务智能体。

执行步骤：

1. 使用 `01-售前一页纸.md` 讲清产品价值和边界。
2. 判断客户是否有稳定 FAQ、服务流程、价格范围和转人工方式。
3. 明确首版采用 Dify WebApp 或嵌入方式上线。
4. 告知 API、CRM、企微、表格同步属于二阶段增强。
5. 使用 `05-报价与套餐模板.md` 给出试点报价范围。

交付物：

- 客户确认的首版范围。
- 客户确认的报价与付款方式。
- 客户确认的资料准备清单。

## 阶段 2：需求采集

目标：把客户业务信息转成可搭建的智能客服需求。

执行步骤：

1. 使用 `02-客户需求采集表.md` 采集业务信息。
2. 确认首版线索字段。
3. 确认必须转人工的问题。
4. 确认上线入口。
5. 收集客户真实验收问题。

交付物：

- 完整需求采集表。
- 原始客户资料。
- 验收问题清单。

## 阶段 3：知识库整理

目标：把客户资料整理成可被 Dify 检索的结构化知识库。

执行步骤：

1. 按 `03-客户知识库整理模板.md` 整理 Markdown。
2. 删除重复、冲突和未经确认的内容。
3. 对价格、合同、退款、投诉内容增加保守边界。
4. 把客户未提供答案的问题列为知识库缺口。
5. 让客户确认最终知识库文本。

交付物：

- 客户知识库 Markdown。
- 知识库缺口清单。
- 客户确认记录。

## 阶段 4：Dify 搭建

目标：搭建可测试的客户专属智能客服业务智能体。

执行步骤：

1. 新建或复制 Dify Chatflow。
2. 上传客户知识库。
3. 配置 Knowledge Retrieval 节点。
4. 配置 LLM 节点和系统提示词。
5. 配置 Answer 节点。
6. 使用 `deepseek-v4-flash` 作为首版主模型。
7. 预留 `deepseek-v4-pro` 作为复杂问题升级模型选择。
8. 发布前完成内部冒烟测试。

交付物：

- Dify 应用。
- 知识库。
- 系统提示词。
- 测试记录。

## 阶段 5：客户演示与修改

目标：让客户看到真实业务效果，并收敛上线前修改。

执行步骤：

1. 用客户验收问题演示。
2. 记录回答不准确、未覆盖或客户不满意的问题。
3. 修改知识库，而不是随意放宽智能体承诺。
4. 对转人工、价格、售后边界做重点复测。
5. 形成上线候选版本。

交付物：

- 演示记录。
- 修改清单。
- 上线候选版本。

## 阶段 6：上线验收

目标：按统一标准确认首版可以交付。

执行步骤：

1. 使用 `06-验收清单.md` 执行验收。
2. 验证 WebApp 或嵌入入口可访问。
3. 验证移动端可打开。
4. 验证 12 个客户问题。
5. 验证转人工和兜底话术。
6. 验证客户能查看对话记录。
7. 发布最终应用。

交付物：

- 验收清单。
- 发布入口。
- 客户培训记录。

## 阶段 7：运维交接

目标：让客户知道如何维护，而不是把智能体当成一次性项目。

执行步骤：

1. 使用 `07-运维维护手册.md` 做培训。
2. 说明知识库更新流程。
3. 说明哪些问题必须交给人工。
4. 说明对话日志如何回看。
5. 约定首次复盘时间。

交付物：

- 运维维护手册。
- 复盘时间。
- 后续增强建议。

## 阶段 8：二阶段 API 接入

触发条件：

- 客户需要自定义网页样式。
- 客户需要隐藏 Dify API Key。
- 客户需要线索同步到表格、CRM、飞书或企微。
- 客户需要统一日志、限流、错误兜底或客户隔离。

执行入口：

- 使用 `08-API接入实施方案.md` 进入二阶段。
