# 多行业客服智能体模板研究

日期：2026-07-07

负责人：学习研究 Agent

阶段：多行业客服智能体模板矩阵 + 标准商业化交付包

## 1. 研究结论

智能客服业务智能体进入复制阶段后，不宜先扩成“大而全行业库”，应优先沉淀 3 个低合规风险、需求高频、资料容易标准化、可用 Dify Chatflow 快速复用的行业模板：

1. 教育培训客服模板。
2. 家政清洁客服模板。
3. 本地门店客服模板。

这 3 类行业共同适合首批模板矩阵，原因是客户问题集中在服务介绍、价格范围、预约流程、位置时间、售后说明和线索收集；不需要首版接入复杂交易系统，也不应承诺结果、资质、医疗效果、升学结果、退款裁定或最终报价。

## 2. 通用模板底座

### 2.1 通用意图分类

所有行业模板建议保留以下意图：

| 意图 | 处理方式 |
| --- | --- |
| 服务咨询 | 检索知识库并回答服务范围、适用对象、流程 |
| 价格咨询 | 只回答价格区间、影响因素和需人工确认的信息 |
| 预约咨询 | 说明预约路径，并进入线索收集 |
| 售后/投诉 | 说明处理流程，必要时转人工 |
| 购买/到店/上门意向 | 分步收集线索字段 |
| 人工请求 | 直接转人工 |
| 未知/超范围 | 不编造，提示人工跟进 |
| 高风险问题 | 拒绝承诺或诊断，转人工 |

### 2.2 通用知识库模块

- 商家简介：品牌、服务区域、营业时间、联系方式口径。
- 服务项目：服务名称、适用对象、服务流程、常见限制。
- 价格说明：价格区间、计费单位、影响价格的因素、最终报价确认方式。
- 预约流程：用户如何预约、需要准备什么、多久有人联系。
- 售后与投诉：处理流程、所需信息、人工介入节点。
- 禁止承诺边界：不能保证的内容、不能替人工判断的内容。
- FAQ：高频问题与标准回答。

### 2.3 通用 Dify 配置要点

- 应用类型：优先使用 Chatflow。
- 节点结构：Start -> Question Classifier -> Knowledge Retrieval -> LLM Answer / Lead Capture / Handoff Answer -> Final Answer。
- 变量保持行业通用，行业差异放到知识库和提示词分支中。
- 每个行业单独维护 Knowledge，避免不同客户资料混杂。
- 线索收集每次只问 1-2 个字段，避免像表单审问。
- 对价格、售后、投诉、合同、资质、效果承诺问题设置强制转人工或保守回答。

## 3. 优先行业模板一：教育培训客服

### 3.1 适用客户

适合本地教培机构、职业技能培训、兴趣班、成人教育咨询、少儿素质课程咨询。不建议首批覆盖涉及强监管或强承诺的升学保过、学历代办、证书包过等业务。

### 3.2 典型客户问题

| 问题类型 | 示例问题 | 推荐处理 |
| --- | --- | --- |
| 课程介绍 | “你们有哪些课程？”“适合几年级？” | 从课程知识库回答适用对象、课时、上课方式 |
| 价格咨询 | “一节课多少钱？”“有没有优惠？” | 给价格区间和影响因素，引导人工确认 |
| 试听/预约 | “可以试听吗？”“怎么报名？” | 进入预约线索收集 |
| 师资咨询 | “老师是不是名师？”“有没有教师资格证？” | 只按资料说明师资背景，不夸大 |
| 效果咨询 | “能保证提分吗？”“能不能包过？” | 禁止承诺结果，转人工说明学习评估 |
| 退费/投诉 | “不满意能退吗？”“我要投诉老师” | 说明流程并转人工 |

### 3.3 知识库模块

- 课程目录：课程名称、适用年龄/年级、课程目标、上课形式。
- 课时与排班：班型、上课时间、线上/线下、校区信息。
- 价格与优惠：价格区间、试听政策、优惠活动有效期。
- 师资介绍：教师背景、授课经验、可公开资质说明。
- 报名流程：试听、测评、报名、缴费、开课通知。
- 售后规则：调课、请假、退费、投诉处理流程。
- 风险边界：不承诺提分、升学、证书、考试结果。

### 3.4 留资字段

| 字段 | 用途 |
| --- | --- |
| 学员称呼 | 后续顾问沟通 |
| 联系方式 | 电话或微信回访 |
| 年级/年龄 | 判断课程匹配 |
| 目标科目/课程 | 匹配课程顾问或校区 |
| 所在城市/校区 | 判断可服务范围 |
| 期望上课时间 | 判断排班 |
| 是否需要试听 | 区分普通咨询和强意向 |

### 3.5 必须转人工场景

- 用户要求最终报价、专属优惠、合同条款。
- 用户要求承诺提分、升学、考试通过、证书获取。
- 涉及退费争议、投诉老师、未成年人安全或纠纷。
- 用户询问具体教师资质证明、机构办学许可等需核验资料。
- 模型无法从知识库确认课程、校区、排班。

### 3.6 禁止承诺边界

- 不承诺“保证提分”“包过”“包拿证”“一定录取”。
- 不代替机构做最终价格、退款、合同、投诉裁定。
- 不编造教师履历、资质编号、合作学校或官方背书。
- 不对未成年人心理、健康、升学路径做专业判断。

### 3.7 可复制配置要点

- Question Classifier 增加 `course_intro`、`trial_booking`、`teacher_info`、`result_promise_risk`。
- Lead Capture 优先收集年级/课程/校区/试听意向，不先追问过多家庭信息。
- 系统提示词固定加入：“学习效果受基础、投入、课程匹配等因素影响，不能做结果承诺。”
- 测试集必须覆盖“能不能保证提高 30 分”“老师是不是名校毕业”“我要退费”等边界问题。

## 4. 优先行业模板二：家政清洁客服

### 4.1 适用客户

适合保洁、家电清洗、开荒清洁、收纳整理、保姆/月嫂初步咨询、搬家前后清洁等本地到家服务。涉及母婴护理、老人护理、医疗护理时只做信息收集和人工转接，不做专业判断。

### 4.2 典型客户问题

| 问题类型 | 示例问题 | 推荐处理 |
| --- | --- | --- |
| 服务范围 | “你们能做深度保洁吗？”“能清洗油烟机吗？” | 回答服务项目和覆盖区域 |
| 价格咨询 | “两室一厅多少钱？”“开荒怎么算？” | 按面积/项目给估算口径，最终报价转人工 |
| 预约上门 | “明天能来吗？”“几点有空？” | 收集地址区域、面积、服务项目、期望时间 |
| 服务人员 | “阿姨有没有健康证？”“能指定人员吗？” | 按知识库说明管理流程，具体名单转人工 |
| 售后问题 | “没清干净怎么办？”“东西损坏怎么办？” | 说明售后流程并转人工 |
| 高风险需求 | “能照顾病人吗？”“能处理危险物品吗？” | 不承诺，转人工确认 |

### 4.3 知识库模块

- 服务项目：日常保洁、深度保洁、开荒清洁、家电清洗、收纳整理。
- 服务区域：城市、区县、是否收上门费。
- 计价规则：按小时、面积、项目、加急、耗材等影响因素。
- 预约流程：下单、确认时间、上门、验收、售后。
- 服务人员管理：培训、着装、证件、服务规范的可公开口径。
- 售后处理：返工、投诉、损坏反馈、证据提交。
- 不服务范围：危险物品、高空外墙、医疗护理、违法违规事项。

### 4.4 留资字段

| 字段 | 用途 |
| --- | --- |
| 称呼 | 客服回访 |
| 联系方式 | 电话或微信确认 |
| 城市/小区或服务区域 | 判断是否可上门 |
| 服务项目 | 匹配服务人员 |
| 房屋面积/户型 | 估算工时和价格 |
| 期望上门时间 | 排班 |
| 特殊说明 | 宠物、重油污、是否有电梯、停车等 |

### 4.5 必须转人工场景

- 用户要求确认当天具体档期、指定服务人员、最终报价。
- 涉及损坏赔付、投诉、返工、退款。
- 涉及老人护理、母婴护理、病患照护等专业或高责任服务。
- 涉及高空、危险品、违法清运、钥匙托管等风险事项。
- 地址、服务内容、面积信息不足但用户要求马上报价。

### 4.6 禁止承诺边界

- 不承诺“绝对无损坏”“一定当天上门”“一定最低价”。
- 不承诺具体服务人员资质，除非知识库有明确可公开资料。
- 不替客户判断安全风险、护理方案或医疗相关事项。
- 不处理或承诺违法违规、危险操作。

### 4.7 可复制配置要点

- Question Classifier 增加 `service_area`、`booking_schedule`、`after_sales_cleaning`、`safety_risk`。
- Lead Capture 重点收集服务项目、面积、区域、时间和特殊情况。
- 价格回答使用“估算口径 + 影响因素 + 人工确认”，不要生成确定金额。
- Handoff Answer 对赔付、投诉、危险服务使用固定转人工话术。

## 5. 优先行业模板三：本地门店客服

### 5.1 适用客户

适合餐饮、烘焙、宠物店、摄影馆、健身工作室、茶饮店、社区零售、维修门店等低风险本地服务。若扩展到医美、口腔、诊所等轻咨询行业，首批只允许做项目介绍、预约和转人工，禁止诊断、疗效和医疗建议。

### 5.2 典型客户问题

| 问题类型 | 示例问题 | 推荐处理 |
| --- | --- | --- |
| 门店信息 | “地址在哪？”“几点营业？” | 回答地址、营业时间、交通信息 |
| 产品/服务 | “有什么套餐？”“能不能预约拍照？” | 回答项目、套餐、流程 |
| 到店/预约 | “今天能约吗？”“怎么排队？” | 收集到店时间、人数/项目、联系方式 |
| 活动咨询 | “有没有团购？”“会员卡怎么办？” | 说明公开活动规则，最终优惠转人工 |
| 售后投诉 | “体验不好怎么办？”“我要退款” | 说明流程并转人工 |
| 高风险咨询 | “这个项目一定有效吗？”“能治疗吗？” | 禁止承诺，转人工 |

### 5.3 知识库模块

- 门店基础信息：地址、营业时间、停车、交通、联系电话。
- 产品/服务清单：项目名称、适用人群、流程、时长。
- 套餐与价格：公开价格、团购规则、会员权益、有效期。
- 预约/排队规则：预约方式、取消改期、到店准备。
- 售后与投诉：退款、改期、体验反馈处理流程。
- 行业边界：餐饮过敏提醒、宠物健康限制、医美/口腔/诊所高风险转人工。

### 5.4 留资字段

| 字段 | 用途 |
| --- | --- |
| 称呼 | 门店客服跟进 |
| 联系方式 | 确认预约 |
| 意向门店/城市 | 多门店分流 |
| 意向项目/套餐 | 匹配服务 |
| 到店人数/宠物类型/项目对象 | 视行业模板启用 |
| 期望到店时间 | 排班或预约 |
| 备注 | 过敏、改期、特殊需求等 |

### 5.5 必须转人工场景

- 用户要求当天实时库存、实时排队、具体技师/摄影师/教练档期。
- 用户要求最终优惠、退款、赔付、投诉处理。
- 涉及过敏、宠物疾病、身体不适、医美效果、诊疗建议。
- 用户要求平台外交易、私下转账、违规服务。
- 知识库没有该门店或该活动的最新信息。

### 5.6 禁止承诺边界

- 不承诺“今天一定有位”“一定能约到指定人员”。
- 不承诺疗效、改善效果、减重结果、变美结果或治疗效果。
- 不替门店裁定退款、赔偿、会员纠纷。
- 不编造活动、库存、档期和门店政策。

### 5.7 可复制配置要点

- 本地门店模板应做成“基础门店客服 + 行业差异包”结构。
- 基础变量保留 `store_location`、`service_item`、`visit_time`、`party_size`、`lead_contact`。
- 医美/口腔/健康相关门店默认开启更严格的 `medical_or_effect_risk` 分类。
- 多门店客户需要知识库按门店分组，回答时先确认用户意向门店。

## 6. 三行业模板矩阵对比

| 维度 | 教育培训 | 家政清洁 | 本地门店 |
| --- | --- | --- | --- |
| 最核心意图 | 课程咨询、试听报名 | 服务咨询、预约上门 | 门店信息、到店预约 |
| 关键留资 | 年级/课程/校区 | 区域/面积/服务项目 | 门店/项目/到店时间 |
| 价格口径 | 课程区间 + 顾问确认 | 面积/项目估算 + 人工确认 | 公开套餐 + 活动人工确认 |
| 高风险边界 | 不承诺提分、升学、证书 | 不承诺安全、护理、赔付 | 不承诺疗效、库存、档期 |
| 强制转人工 | 退费、投诉、结果承诺 | 赔付、危险服务、护理 | 退款、投诉、健康/疗效 |
| 复制难度 | 中 | 低 | 低到中 |
| 首批推荐等级 | P0 | P0 | P1 |

## 7. 标准商业化交付包中的研究输入

学习研究侧建议把每个行业模板交给后续角色时，至少提供以下材料：

1. 行业 FAQ 样例：不少于 30 条，覆盖服务、价格、预约、售后、边界。
2. 行业知识库目录：可直接让客户补资料的空表结构。
3. 行业系统提示词边界：尤其是禁止承诺和转人工规则。
4. 行业测试问题集：包含正常咨询、强意向、未知问题、高风险问题。
5. 留资字段清单：区分必填、选填、人工补充字段。
6. 转人工话术：普通转人工、高风险转人工、投诉转人工分别配置。

## 8. 下一步建议

学习研究 Agent 的下一步不直接搭建 Dify 应用，而是继续沉淀模板资料：

1. 为教育培训、家政清洁、本地门店分别补一版 FAQ 样例。
2. 把禁止承诺边界拆成可复制的系统提示词片段。
3. 把测试问题整理给 Agent 搭建师，用于 Dify Chatflow 验收。
4. 若后续扩展医美轻咨询，只能作为“高风险轻咨询模板”单独研究，默认不进入首批 P0。
