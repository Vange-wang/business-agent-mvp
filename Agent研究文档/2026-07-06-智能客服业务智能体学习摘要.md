# 智能客服业务智能体学习摘要

日期：2026-07-06

任务 ID：`CS-AGENT-MVP-2026-07-06-001-RESEARCH`

负责人：学习研究 Agent

落盘执行：项目总负责人前台线程

## 1. 学习结论

智能客服业务智能体不应被做成“聊天机器人壳子”，而应围绕真实商家客服闭环设计：知识库问答、意图判断、线索收集、转人工和运营复盘。

首版应优先用 Dify 完成可复制流程，Coze 作为插件、工作流和交互体验参考。

## 2. 可复用做法

- 先整理商家 FAQ，再搭建智能体。
- 知识库内容按“服务介绍 / 价格 / 流程 / 售后 / 转人工”分块。
- 使用意图分类区分普通问答、购买意向和高风险问题。
- 对购买意向用逐步追问，不一次性索要过多信息。
- 对无法回答的问题固定转人工，不编造答案。
- 用测试问题集检查知识库覆盖率。

## 3. 不适合照搬的做法

- 直接复制教程里的虚构行业。
- 把模型说得像能完全替代人工。
- 未确认客户资料就写复杂 CRM 接入。
- 首版就做多租户后台、支付和自动计费。
- 将 Dify / Coze 平台能力包装成自己的完全自研平台。

## 4. 推荐流程

1. 客户访客进入客服入口。
2. 智能客服识别问题类型。
3. 普通问题走知识库检索。
4. 购买/预约意向走线索收集。
5. 高风险或无答案问题走转人工。
6. 对话记录用于后续知识库优化。

## 5. FAQ/RAG 搭建要点

- 每条 FAQ 应有明确问题和标准答案。
- 价格只给范围，不给最终承诺。
- 售后和退款只给流程，不代替人工决定。
- 知识库缺失要记录为后续补充项。

## 6. 线索收集要点

建议字段：

- 称呼。
- 联系方式。
- 需求类型。
- 城市/区域。
- 期望联系时间。
- 备注。

追问方式：

- 先确认用户意向。
- 再说明收集信息的用途。
- 每次只问 1-2 个字段。

## 7. 转人工要点

转人工触发：

- 用户明确要求人工。
- 模型不确定。
- 投诉、退款、合同、开票、正式报价。
- 用户输入个人隐私或高风险内容。

## 8. Dify vs Coze 学习侧建议

- Dify：主线，适合知识库、工作流、API、日志和私有化交付。
- Coze：参考其插件、变量、工作流交互和多步骤任务体验。
- 首版不要同时维护两套平台实现，避免交付复杂度翻倍。
