# 多行业客服智能体模板矩阵总览

## 阶段定位

本目录用于沉淀“多行业客服智能体模板矩阵 + 标准商业化交付包”的 Dify 搭建材料。当前文件面向内部 Agent 搭建师使用，不是客户宣传页，也不是 Dify 官方导出 DSL。

本阶段目标是把已经验证过的智能客服 MVP 和第二客户复制试点，扩展为可快速复制到不同行业的模板包。每个行业模板都要能直接复制到 Dify 的应用说明、系统提示词、开场白、建议问题、知识库整理和测试验收环节中使用。

## 模板矩阵

| 行业模板 | 适用客户 | 核心目标 | 推荐 Dify 类型 | 线索价值 |
| --- | --- | --- | --- | --- |
| 教育培训客服模板 | 教培机构、素质教育、职业培训、升学咨询 | 课程咨询、试听预约、校区/线上咨询转人工 | Chatflow + Knowledge | 学员姓名、年龄/年级、课程方向、联系方式 |
| 家政清洁客服模板 | 家政公司、保洁门店、上门服务团队 | 服务说明、价格范围、上门预约、复杂需求转人工 | Chatflow + Knowledge | 服务地址、房屋面积、服务类型、期望时间 |
| 本地门店客服模板 | 餐饮、美容美发、健身、维修、小零售门店 | 营业信息、预约到店、活动咨询、投诉售后转人工 | Chatflow + Knowledge | 门店、到店时间、服务/商品需求、联系方式 |

## 统一 Dify 搭建结构

### 应用命名

建议命名格式：

`客户简称-行业客服智能体-v1`

示例：

- `星桥教育-课程咨询客服-v1`
- `洁家到家-家政清洁客服-v1`
- `南街门店-到店咨询客服-v1`

### 推荐节点

1. 开始节点：接收用户问题。
2. 知识库检索节点：检索行业 FAQ、服务规则、价格范围、预约流程。
3. LLM 回答节点：基于知识库回答，限制禁答边界。
4. 线索识别节点：识别预约、咨询、购买、试用、到店意向。
5. 留资收集节点：分步收集关键字段，不一次性索要过多信息。
6. 转人工节点：命中高风险、投诉、退款、合同、最终报价等条件时转人工。
7. 结束节点：确认已记录需求，说明后续人工联系或门店跟进。

### 统一系统提示词底线

可复制到所有行业模板的基础规则：

```text
你是商家的智能客服业务智能体。你的目标是接管重复咨询，基于知识库回答问题，识别有效意向，分步收集线索，并在无法确定或高风险场景中转人工。

必须遵守：
1. 只能基于知识库和用户已提供信息回答，不编造价格、政策、承诺、资质和库存。
2. 回答要简洁、自然、可信，优先给用户下一步行动。
3. 涉及最终报价、退款、投诉、合同、开票、法律责任、医疗诊断、效果保证、隐私数据等问题，必须转人工。
4. 当用户表达预约、报名、购买、到店、试用、报价、上门、联系意向时，分步收集称呼、联系方式、需求类型、所在城市/区域、期望联系时间。
5. 不要一次性索要所有信息。每次最多询问 2-3 个字段。
6. 如果知识库没有答案，回复：这个问题需要人工进一步确认，我可以先帮你记录需求。
7. 不承诺“完全替代人工”，不代表商家做最终决策。
```

## 统一交付检查清单

上线前至少完成以下检查：

1. 知识库已填入真实客户资料，未混入其他客户或测试隐私数据。
2. 应用开场白、建议问题已替换为客户行业话术。
3. 系统提示词包含禁答边界和转人工触发条件。
4. 留资字段能覆盖客户后续跟进所需信息。
5. 验收问答样例全部通过：FAQ 可回答、未知问题不编造、价格只给范围、投诉退款转人工、预约留资正常。
6. 客户已确认人工接收方式，例如企业微信、电话、表单、CRM、飞书/钉钉/企微群或 Webhook。

## 标准商业化交付包建议

每个客户交付时至少准备：

1. Dify 应用配置截图或配置说明。
2. 行业知识库 Markdown。
3. 系统提示词版本记录。
4. 验收问答记录。
5. 客户培训说明：如何新增 FAQ、如何查看对话、如何接收线索、如何触发人工。
6. 维护边界说明：知识库更新、提示词优化、渠道接入、异常问题处理。

## 不纳入当前模板矩阵的范围

当前阶段不默认包含：

- 多租户 SaaS 后台。
- 支付、自动计费、发票系统。
- 私有模型微调。
- 客户生产数据迁移。
- 大规模插件市场。
- 对外承诺完全无人客服。
