# 本地门店 Dify 实装配置蓝图

## 状态说明

本文是本地门店行业的 Dify 实装配置蓝图和验收材料，不代表已经登录 Dify 完成线上应用。搭建师可按本文在 Dify 中新建 Chatflow、配置 Knowledge、变量、节点和测试批次。

## Dify 应用配置

- 应用名称：`客户简称-本地门店到店助手-v1`
- 应用类型：Chatflow
- 绑定知识库：`客户简称-本地门店知识库-v1`
- 应用描述：用于回答营业时间、门店地址、服务/商品、活动、预约和到店流程，收集到店预约或购买意向，并把实时库存、档期、投诉、退款等问题转人工。

## 知识库配置

知识库名称：`客户简称-本地门店知识库-v1`

建议拆分为 6 个文档：

1. `门店基础信息.md`：门店简介、主营服务/商品、地址、交通、停车、营业时间。
2. `服务与商品.md`：服务项目、商品类型、公开价格范围、购买方式。
3. `预约与到店流程.md`：预约规则、取消规则、迟到规则、到店流程。
4. `活动与会员.md`：当前有效活动、适用条件、会员公开规则；过期活动必须删除。
5. `售后与门店规则.md`：投诉、退款、储值卡、发票、售后处理入口。
6. `禁答与高风险边界.md`：医疗美容、药品、健康诊断、最终价格、实时库存、员工隐私。

## 建议开场白

```text
你好，我是门店咨询助手。可以帮你了解营业时间、地址、服务项目、活动和预约流程。请问你想咨询哪项服务或什么时间到店？
```

## 建议问题

1. 你们今天营业吗？
2. 门店地址在哪里？
3. 可以预约明天到店吗？
4. 最近有什么活动？
5. 你们有哪些服务/商品？
6. 停车方便吗？
7. 我想让门店联系我。

## 核心节点配置

| 节点名称 | Dify 节点类型 | 配置要点 |
| --- | --- | --- |
| Start | 开始 | 接收用户输入，保留会话上下文。 |
| ExtractStoreIntent | LLM / 参数提取 | 输出 `intent_type`：营业时间、地址、服务商品、活动、预约、库存、投诉、退款、未知。 |
| ExtractStoreLead | 参数提取 | 抽取服务/商品、到店时间、门店/区域、联系方式、人数、需求细节。 |
| StoreKnowledgeRetrieve | 知识检索 | 绑定 `客户简称-本地门店知识库-v1`，检索地址、营业时间、活动、预约规则。 |
| StoreRiskBranch | 条件分支 | 命中退款、投诉、储值卡、合同、医疗美容、实时库存/档期确认时进入转人工。 |
| StoreLeadBranch | 条件分支 | 用户表达预约、到店、购买、咨询活动、想联系门店时进入留资。 |
| StoreAnswer | LLM | 基于知识库回答，不编造库存、档期、座位、技师、优惠。 |
| StoreCollectLead | LLM | 每轮最多询问 2-3 个字段，优先收集服务/商品、到店时间、联系方式。 |
| StoreHumanHandoff | LLM / HTTP 请求占位 | 输出门店人工跟进摘要；未接系统前先让用户留下联系方式。 |
| End | 结束 | 告知已记录需求，后续由门店确认实时情况。 |

## 变量配置

| 变量 | 类型 | 必填 | 用途 |
| --- | --- | --- | --- |
| `intent_type` | enum | 是 | `hours` / `address` / `service_product` / `promotion` / `booking` / `stock` / `complaint` / `refund` / `risk` / `unknown`。 |
| `risk_flag` | boolean | 是 | 是否必须转人工。 |
| `user_name` | string | 否 | 用户称呼。 |
| `phone_or_wechat` | string | 是 | 留资最小必填字段。 |
| `store_or_area` | string | 否 | 指定门店、城市或区域。 |
| `service_or_product` | string | 是 | 想预约的服务或咨询的商品。 |
| `visit_time` | string | 否 | 期望到店日期和时间。 |
| `people_count` | string | 否 | 到店人数或服务人数。 |
| `demand_detail` | string | 否 | 具体需求、预算、偏好。 |
| `need_summary` | string | 是 | 自动生成给门店员工的摘要。 |
| `handoff_reason` | string | 否 | 转人工原因。 |

## 留资字段

最小可交付线索：

1. 称呼：`user_name`
2. 联系方式：`phone_or_wechat`
3. 服务/商品需求：`service_or_product`
4. 门店或区域：`store_or_area`

增强字段：

1. 到店时间：`visit_time`
2. 人数：`people_count`
3. 需求细节：`demand_detail`
4. 特殊说明：写入 `need_summary`

留资话术：

```text
可以，我先帮你记录预约意向。请问你想预约哪项服务、预计什么时候到店？方便的话留下手机号或微信，门店会再确认档期。
```

## 转人工触发

命中以下任一情况，`risk_flag=true` 并进入 `StoreHumanHandoff`：

1. 实时库存、座位、档期、技师、房间或商品余量确认。
2. 最终价格、套餐变更、会员储值、退款、转卡。
3. 投诉服务、商品质量、门店体验。
4. 合同、开票、发票抬头、售后赔偿。
5. 医疗美容、药品、健康诊断、高风险操作。
6. 用户已留下联系方式并要求门店联系。
7. 知识库没有当前活动、营业变更或门店信息。

转人工话术：

```text
这个需要门店人工确认实时情况。我可以先帮你记录需求，请留下称呼、联系方式、想预约的项目和预计到店时间。
```

## 验收测试批次

### 批次 1：基础 FAQ

| 测试问题 | 期望结果 |
| --- | --- |
| 你们今天营业吗？ | 基于营业时间回答；节假日或临时调整不确定则转人工。 |
| 门店在哪里？ | 回答地址、地标、交通或停车信息。 |
| 你们有哪些服务/商品？ | 基于知识库列主要项目，并询问用户具体需求。 |

### 批次 2：到店预约留资

| 测试问题 | 期望结果 |
| --- | --- |
| 明天下午可以预约吗？ | 收集服务项目、到店时间、人数/需求、联系方式，说明门店确认档期。 |
| 我想让门店联系我。 | 收集称呼、联系方式、服务/商品、到店时间和门店/区域。 |

### 批次 3：活动与库存

| 测试问题 | 期望结果 |
| --- | --- |
| 最近有什么优惠？ | 只回答知识库中有效活动；不确定是否过期则转人工。 |
| 这个商品现在有货吗？ | 实时库存需人工确认，引导留下联系方式和商品需求。 |

### 批次 4：风险转人工

| 测试问题 | 期望结果 |
| --- | --- |
| 我要退款。 | 直接转人工，不自行判断退款规则。 |
| 可以开发票吗？ | 转人工确认开票规则和信息。 |
| 做这个项目有没有风险？ | 普通服务按知识库说明；医疗/健康/高风险问题必须转人工。 |

### 批次 5：未知问题

| 测试问题 | 期望结果 |
| --- | --- |
| 你们能不能上门服务？ | 如知识库无内容，说明需门店人工确认并记录需求。 |
| 你们和某某品牌是不是联名？ | 只按知识库回答；没有则转人工，不编造合作。 |
