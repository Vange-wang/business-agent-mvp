# 三行业 Dify 实装配置蓝图与验收总览

## 阶段定位

本目录是“多行业 Dify 模板矩阵”的下一层实装配置蓝图和验收材料，面向内部 Agent 搭建师使用。

当前状态必须明确：

- 这里不是已经登录 Dify 完成的三个线上应用。
- 这里不是 Dify 官方 DSL 导出文件。
- 这里是搭建师可照着在 Dify 中复制、配置、测试和验收的行业实装说明。

## 三行业实装对象

| 行业 | 建议 Dify 应用名称 | 建议知识库名称 | 推荐应用类型 | 主要成交动作 |
| --- | --- | --- | --- | --- |
| 教育培训 | `客户简称-教育培训咨询助手-v1` | `客户简称-教育培训知识库-v1` | Chatflow + Knowledge | 课程咨询、试听预约、测评预约、顾问跟进 |
| 家政清洁 | `客户简称-家政清洁预约助手-v1` | `客户简称-家政清洁知识库-v1` | Chatflow + Knowledge | 服务咨询、上门预约、报价线索、客服跟进 |
| 本地门店 | `客户简称-本地门店到店助手-v1` | `客户简称-本地门店知识库-v1` | Chatflow + Knowledge | 到店预约、活动咨询、商品/服务咨询、门店跟进 |

## 统一搭建顺序

1. 在 Dify 新建 Chatflow 应用，按行业文件填写应用名称、描述、开场白和建议问题。
2. 新建对应 Knowledge，按行业文件拆分知识库章节并上传 Markdown 或文档。
3. 在 Chatflow 中配置开始节点、变量提取节点、知识库检索节点、条件分支节点、LLM 回答节点、留资节点、转人工节点和结束节点。
4. 把行业文件中的禁答边界写入系统提示词或 LLM 节点指令。
5. 配置留资字段，先用 Dify 变量承接；后续如接 CRM、企微、飞书、表单或 Webhook，再交给代码开发员处理。
6. 按行业验收测试批次逐条测试，记录“通过 / 需改知识库 / 需改提示词 / 需转人工规则”。

## 统一节点骨架

| 节点 | Dify 节点类型建议 | 作用 |
| --- | --- | --- |
| Start | 开始 | 接收用户问题、渠道信息和历史上下文。 |
| ExtractIntent | LLM / 参数提取 | 识别用户意图：FAQ、价格、预约、投诉、退款、未知问题、转人工。 |
| ExtractLeadFields | 参数提取 | 从对话中抽取称呼、联系方式、需求类型、区域、时间等字段。 |
| KnowledgeRetrieve | 知识检索 | 检索行业知识库，优先回答服务、课程、门店、流程和 FAQ。 |
| RiskBranch | 条件分支 | 判断是否命中最终报价、退款、投诉、合同、隐私、承诺等风险。 |
| LeadBranch | 条件分支 | 判断是否已有明确预约、报名、购买、上门、到店或联系意向。 |
| AnswerWithKB | LLM | 基于知识库回答，不编造，不越权承诺。 |
| CollectLead | LLM / 表单 / 参数提取 | 分步收集缺失字段，每次最多问 2-3 个字段。 |
| HumanHandoff | LLM / HTTP 请求占位 | 生成转人工摘要；未接外部系统前先输出人工跟进信息。 |
| End | 结束 | 告知已记录需求、后续由人工确认。 |

## 统一变量命名

| 变量 | 类型 | 通用说明 |
| --- | --- | --- |
| `user_message` | string | 用户当前输入。 |
| `intent_type` | enum | `faq` / `price` / `booking` / `lead` / `complaint` / `refund` / `risk` / `unknown`。 |
| `risk_flag` | boolean | 是否命中必须转人工的风险场景。 |
| `lead_ready` | boolean | 是否已具备可交给人工跟进的最小线索。 |
| `user_name` | string | 用户称呼。 |
| `phone_or_wechat` | string | 手机号或微信。 |
| `need_summary` | string | 用户需求摘要。 |
| `city_or_area` | string | 城市、区域、校区、门店或服务范围。 |
| `preferred_time` | string | 期望试听、上门、到店或人工联系时间。 |
| `handoff_reason` | string | 转人工原因。 |

## 统一验收批次

每个行业至少跑 5 批测试：

1. FAQ 批次：基础问题能基于知识库回答。
2. 价格批次：能给范围和影响因素，但不输出最终报价。
3. 留资批次：能分步收集最小必要字段，不一次性索要过多信息。
4. 转人工批次：退款、投诉、合同、最终报价、效果承诺等场景必须转人工。
5. 未知问题批次：知识库没有答案时不编造，能引导记录需求或转人工。

## 交付边界

当前目录完成后，搭建师可以开始按蓝图配置 Dify 应用；但真实上线前仍需要客户提供或确认：

- 真实课程、服务、门店、价格范围、活动、营业时间和服务区域。
- 人工接收线索的渠道，例如企微、飞书、钉钉、CRM、表单或 Webhook。
- 投诉、退款、合同、开票、售后等人工处理规则。
- 是否允许在客户真实渠道嵌入 Dify WebApp 或 API。
