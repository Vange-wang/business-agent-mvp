# 智能客服业务智能体代码集成准备

任务 ID：`CS-AGENT-MVP-2026-07-06-001-CODE-PREP`

日期：2026-07-06

角色：代码开发员 Agent

服务对象：第一个可售卖业务智能体 MVP - 智能客服业务智能体

## 1. 当前结论

首版 MVP 暂不需要立即编写复杂业务代码。当前更稳妥的代码侧路线是：

1. 先由产品经理确认 Spec、业务范围、验收标准和客户交付口径。
2. 由技术验证 Agent 确认 Dify 本地或自托管部署、模型接入、知识库、API、日志和安全边界。
3. 由 Agent 搭建师在 Dify 中完成智能客服业务智能体、知识库、Chatflow / Workflow、转人工口径和线索收集流程。
4. 代码开发员在以上三项明确后，再补齐嵌入页、API 调用示例、环境变量模板、部署脚本和测试脚本。

若 Dify Web App / Embedding 已满足首版演示和客户试点需求，首版可以优先采用 Dify 官方 WebApp 或嵌入方式交付；自定义 API 中转、客户网站组件、线索落库、后台管理页和自动化脚本应作为后续实现项，不在 Spec 确认前提前扩大范围。

## 2. Dify 集成路径

### 2.1 首版优先路径：Dify WebApp / 嵌入

适用条件：

- 客户只需要网页入口、独立访问链接或嵌入到现有站点。
- 首版主要验证 FAQ、产品资料、服务流程问答、销售线索收集和转人工口径。
- 暂不需要接入客户 CRM、客服系统、工单系统、微信生态或私有业务后台。

代码侧准备内容：

- 记录 Dify 应用发布链接或嵌入配置。
- 准备客户网站嵌入说明。
- 准备嵌入页最小 HTML / Next.js 页面方案。
- 准备测试清单：加载、会话、移动端显示、基础问答、线索收集、失败兜底。

### 2.2 后续增强路径：API 调用

适用条件：

- 客户需要自定义前端样式、统一品牌入口或接入已有网站。
- 需要由服务端隐藏 Dify API Key。
- 需要将销售线索同步到表格、CRM、工单、数据库或消息通知。
- 需要统一日志、限流、客户隔离或错误兜底。

代码侧准备内容：

- 服务端 API 中转示例。
- 前端聊天组件或嵌入页。
- 环境变量模板。
- API 错误处理和超时处理。
- 测试脚本与冒烟检查。

## 3. 前端嵌入页准备

首版嵌入页应保持最小可交付，不引入客户后台、登录系统、支付系统或多租户能力。

建议内容：

- 独立演示页：展示客户品牌名、智能客服入口、服务时间、转人工说明。
- 嵌入容器：承载 Dify WebApp iframe 或后续自定义聊天组件。
- 移动端适配：确保手机端入口可点击、聊天窗口不遮挡核心内容。
- 基础配置：支持通过环境变量或配置文件切换 Dify 应用地址。
- 隐私提示：提醒用户不要提交敏感个人信息，最终文案由产品经理和客户确认。

后续如采用 Next.js + TypeScript + React，建议文件结构预留为：

```text
Code文档/app/customer-service/page.tsx
Code文档/features/customer-service/
Code文档/lib/dify/
Code文档/scripts/
```

以上只是预留方向，实际路径需等仓库代码结构和 Spec 确认后再落地。

## 4. API 调用准备

若后续采用 Dify API，需要准备以下最小调用能力：

| 能力 | 说明 | 是否首版必需 |
| --- | --- | --- |
| 发起聊天请求 | 从前端提交用户问题，由服务端调用 Dify | 视嵌入方式决定 |
| 会话 ID 管理 | 保存 Dify conversation_id，支持连续对话 | API 路线必需 |
| 错误兜底 | Dify 不可用、模型超时、网络失败时返回友好提示 | API 路线必需 |
| API Key 保护 | Dify API Key 只放服务端环境变量，不暴露给浏览器 | API 路线必需 |
| 线索字段收集 | 姓名、联系方式、需求类型、备注 | 等产品 Spec 确认 |
| 线索同步 | 同步到表格、CRM、数据库或通知渠道 | 后续增强 |
| 日志记录 | 记录请求时间、错误类型、响应状态 | 自托管或商业交付建议具备 |

API 示例应等待 Dify 应用类型、接口路径、认证方式和部署域名确认后再实现。

## 5. 环境变量与 `.env.example` 准备

后续需要准备 `.env.example`，只写占位符，不写真实密钥。

建议变量：

```text
DIFY_API_BASE_URL=
DIFY_APP_ID=
DIFY_API_KEY=
DIFY_WEBAPP_URL=
DIFY_CONVERSATION_MODE=
CUSTOMER_SERVICE_AGENT_NAME=
PUBLIC_CUSTOMER_SERVICE_ENTRY_URL=
LEAD_WEBHOOK_URL=
LEAD_WEBHOOK_SECRET=
REQUEST_TIMEOUT_MS=
ENABLE_LEAD_CAPTURE=
ENABLE_DEBUG_LOG=
```

变量说明：

- `DIFY_API_BASE_URL`：Dify API 服务地址。
- `DIFY_APP_ID`：Dify 应用标识，是否需要取决于后续接口方案。
- `DIFY_API_KEY`：服务端调用 Dify 的密钥，禁止提交真实值。
- `DIFY_WEBAPP_URL`：Dify WebApp 或嵌入地址。
- `DIFY_CONVERSATION_MODE`：预留聊天模式配置，例如 webapp、api。
- `CUSTOMER_SERVICE_AGENT_NAME`：客户可见的智能客服名称。
- `PUBLIC_CUSTOMER_SERVICE_ENTRY_URL`：客户访问入口或嵌入页地址。
- `LEAD_WEBHOOK_URL`：后续线索同步 Webhook 地址。
- `LEAD_WEBHOOK_SECRET`：Webhook 校验密钥，禁止提交真实值。
- `REQUEST_TIMEOUT_MS`：API 请求超时时间。
- `ENABLE_LEAD_CAPTURE`：是否启用线索收集。
- `ENABLE_DEBUG_LOG`：是否启用调试日志。

## 6. 部署脚本准备

首版部署脚本是否需要实际编写，取决于技术验证 Agent 对部署方式的结论。

可能需要的脚本：

| 脚本 | 用途 | 前置条件 |
| --- | --- | --- |
| `scripts/check-env` | 检查必要环境变量是否存在 | `.env.example` 确认 |
| `scripts/smoke-dify-api` | 冒烟测试 Dify API 是否可访问 | Dify API Key 和测试应用确认 |
| `scripts/smoke-webapp` | 检查 WebApp / 嵌入页可访问 | 发布地址确认 |
| `scripts/backup-dify-notes` | 记录备份检查步骤或调用备份命令 | 部署方案确认 |
| `scripts/deploy-preview` | 部署预览环境 | 前端技术栈确认 |

当前阶段只记录准备项，不创建实际部署脚本，避免在部署方案未确认时引入错误路径。

## 7. 测试脚本与测试点准备

后续测试应覆盖“可访问、可问答、可收集、可兜底、可保护密钥”五类。

### 7.1 嵌入页测试点

- 页面能打开。
- Dify WebApp 或聊天入口能加载。
- 移动端宽度下不遮挡关键按钮。
- 刷新页面后入口仍可用。
- Dify 服务不可用时有可理解提示。

### 7.2 API 调用测试点

- 未配置 API Key 时拒绝启动或返回明确错误。
- API Key 不出现在前端代码和浏览器响应中。
- 普通问题能获得 Dify 回复。
- 多轮会话能延续 conversation_id。
- Dify 超时或错误时返回兜底文案。
- 输入为空、过长或包含明显无效内容时有校验。

### 7.3 线索收集测试点

- 线索字段与产品 Spec 一致。
- 未经确认不强制收集非必要隐私信息。
- 联系方式格式校验符合客户场景。
- 同步失败时不丢失用户可见反馈。
- 测试数据与真实客户数据隔离。

### 7.4 交付测试证据

- 嵌入页截图或访问记录。
- Dify 应用配置截图或导出说明。
- API 冒烟测试输出。
- 线索收集测试样例。
- 错误兜底测试记录。

## 8. 暂不引入的范围

在产品经理 Spec、技术验证报告和搭建师方案确认前，代码侧不默认引入：

- 多租户客户后台。
- 支付、套餐、自动计费。
- 客户登录和权限系统。
- 私有模型微调。
- 大规模插件市场。
- 客户 CRM 深度双向同步。
- 微信、企微、抖音、小红书等平台级接入。
- 真实客户资料、真实 API Key、真实生产数据库。

如后续业务确实需要，应由产品经理明确纳入 Spec，并由技术验证 Agent 先确认可行性和风险。
