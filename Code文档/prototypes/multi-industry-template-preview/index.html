面向内部交付总览

做了什么

把智能客服 MVP 和二客户复制试点沉淀为三行业模板矩阵：教育培训、家政清洁、本地门店，并补齐知识库结构、提示词边界、留资字段和验收问答方向。

为什么这么做

后续客户交付不能每次从零写需求、配置 Dify、整理 FAQ 和验收问题。先形成低风险行业模板，才能复制交付、控制边界、减少知识库污染。

目前程度

文档、模板、三行业 FAQ 测试集与配置蓝图已形成；商业化交付包已覆盖售前、交付清单、上线验收和售后维护。真实 Dify 应用复制、调试窗口跑测和线上发布尚未执行。

交付资产

已经沉淀的可复用材料
  • 1多行业模板矩阵 Spec：阶段目标、客户范围、验收标准和不做事项。
  • 2三行业 Dify 模板：知识库结构、系统提示词、开场白、建议问题和留资字段。
  • 3技术验证蓝图：Dify 复制路径、RAG 检查、三种接入方式和回滚方案。
  • 4商业化复制包：客户演示话术、交付清单、上线验收表和售后维护 SOP。
  • 5三行业 FAQ 测试集：教育培训、家政清洁、本地门店各 30 条，覆盖 FAQ、价格、预约、售后、未知和高风险边界。
  • 6静态可视化预览：把当前交付范围、进度、分工和下一步做成可打开页面。

下一步

从沉淀进入线上样板
  1. 1Agent 搭建师在 Dify 中复制或创建 3 个行业样板应用，替换 Knowledge、应用名称、欢迎语和提示词。
  2. 2技术验证 Agent 使用 90 条固定测试集验证 FAQ、同义改写、流程追问、线索收集和风险转人工。
  3. 3UI / 交付设计 Agent 将可视化页与套餐包整理为客户可读演示材料。
  4. 4代码开发员 Agent 仅在需要嵌入页、API 中转、Webhook 或自动化测试时继续开发。
  5. 5ISSUE 管理员在真实 Dify 实装阶段只登记实际阻塞和验收缺陷。

交付口径：客户购买的是基于客户业务资料搭建、可嵌入真实业务入口、能处理重复咨询和线索收集的客服智能体交付结果；不是 Dify / Coze 平台本身，也不是内部 Codex 协作角色。
