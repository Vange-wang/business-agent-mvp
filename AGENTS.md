# AGENTS.md

本项目是“Agent智能体”产品化交付仓库。协作方式参考 `家教对接website` 项目的多 Agent 运转逻辑，但业务主题调整为：基于 Dify / Coze 等开源或可自托管平台，制作可上线、可售卖、可复制交付给真实客户和真实商家的业务智能体产品。

## 术语边界

本项目必须严格区分两类 Agent：

1. **内部协作角色 Agent**：指项目总负责人、产品经理、技术验证、学习研究、Agent 搭建师、代码开发员、UI / 交付设计、ISSUE 管理员等 Codex 协作线程角色。它们只服务于项目管理、研发、验证和交付协同。
2. **可售卖业务智能体 Agent**：指最终面向客户交付的商业化智能体产品，例如客服 Agent、信息回复 Agent、销售线索收集 Agent、私域运营 Agent、企业知识库问答 Agent 等。它们要接入或适配客户已有软件、平台和业务流程，接管商家重复性动作和重复性工作。

Dify / Coze 是实现、验证和交付业务智能体的工具或底座，不是本项目的最终产品本身；Codex 子会话也不是要卖给客户的业务智能体。

## 当前总目标

1. 建立一套可复用的业务智能体交付流程。
2. 先完成平台选型、技术验证、模板沉淀和商业化边界确认。
3. 后续按客户场景复制：需求调研、知识库整理、业务智能体搭建、测试验收、部署交付、运维迭代。

## 当前结论

- 主平台：Dify。
- 备选与学习平台：Coze Studio / Coze。
- 原因：Dify 更适合自托管、客户数据隔离、RAG 知识库、API 集成、生产观测和可复制交付；Coze 适合学习智能体搭建体验、插件和工作流交互设计，但公开部署和二开落地需要更谨慎。

## 角色分工

- 项目总负责人 Agent：负责多 Agent 编排、任务分发、协同记录、责任溯源和阶段闭环。
- 产品经理 Agent：负责客户场景拆解、业务智能体产品定义、MVP 范围、验收标准和客户交付说明。
- 技术验证 Agent：负责 Dify / Coze / 模型 / RAG / 部署 / API 可行性验证。
- Agent 搭建师 Agent：负责在 Dify 中搭建工作流、知识库、提示词、工具调用和应用模板。
- 代码开发员 Agent：负责自定义插件、业务系统集成、前端嵌入页、自动化脚本和部署配置。
- UI / 交付设计 Agent：负责客户演示页、嵌入入口、交付文档视觉、模板包装和演示材料。
- ISSUE 管理员 Agent：负责问题登记、Open / Close 状态、阻塞归档和关闭依据。
- 学习研究 Agent：负责视频教程、官方文档、竞品案例的学习摘要和可复用方法提炼。

## 文档优先级

1. `规划文档/Spec文档/`：定义客户业务智能体产品做什么、为什么做、如何验收。
2. `规划文档/里程碑文档/`：定义阶段目标、交付范围和验收口径。
3. `规划文档/技术验证/`：记录 Dify / Coze / 模型 / 部署 / 集成验证结论。
4. `Agent研究文档/`：记录视频学习、竞品学习、案例拆解和模板沉淀。
5. `Code文档/`：记录代码、插件、部署脚本、嵌入页和运维说明。
6. `协同工作文档/`：记录 Agent 注册、交流、Issue 和闭环流程。
7. `AGENTS.md`：定义项目协作规则和工作方式。

## MVP 原则

首个可售卖版本不追求“大而全”，优先打通一个可交付闭环：

1. 明确一个高频客户场景，例如客服问答、销售线索收集、私域运营助手、企业知识库问答或短视频脚本助手。
2. 整理客户知识资料和标准业务流程。
3. 在 Dify 中搭建可售卖业务智能体、Chatflow、Workflow 和 Knowledge。
4. 接入一个主流模型和一个备用模型。
5. 提供网页嵌入或 API 调用入口。
6. 完成测试、交付文档、客户培训和后续维护方案。

在 Spec 确认前，不默认加入复杂多租户、支付系统、客户后台、自动计费、私有模型微调或大规模插件市场。

## 权限运行规则

- 只读线程只做阅读、分析、审查和回报。
- 写文件、安装依赖、运行构建、提交、推送、部署等动作，只能在用户确认完全访问的前台线程执行。
- 不提交真实 API Key、客户资料、模型密钥、数据库密码或任何生产隐私数据。
