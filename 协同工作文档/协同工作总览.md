## 2. 协作原则

- 本项目中的“Agent”必须区分两层含义：内部协作角色 Agent 是 Codex 协作线程；可售卖业务智能体 Agent 是面向真实客户和商家交付的产品。
- 后续任务分发中，凡是指 Codex 线程，统一写“内部协作角色 Agent”；凡是指客户交付物，统一写“业务智能体”或“可售卖业务智能体 Agent”。
- 每个会话线程对应一个专业 Agent。
- 每个 Agent 先注册身份，再接收正式任务。
- 每个任务必须明确任务 ID、输入材料、输入边界、输出要求、输出边界、验收标准和阻塞回报格式。
- 项目总负责人负责调度和闭环，不替专业 Agent 越权产出。
- 只读后台线程只负责学习、分析、审查和回报；写入、验证、提交、部署集中到用户确认的完全访问前台线程。

## 3. 本项目推荐角色

| 角色 | 主责 |
| --- | --- |
| 产品经理 Agent | 客户场景、MVP、业务智能体产品定义、验收标准 |
| 技术验证 Agent | Dify / Coze / 模型 / RAG / 部署 / 安全可行性 |
| 学习研究 Agent | 视频教程与竞品资料学习，形成可复用摘要 |
| Agent 搭建师 Agent | Dify 业务智能体工作流、知识库、提示词、工具调用、模板搭建 |
| 代码开发员 Agent | 插件、API、嵌入页、部署脚本和自动化 |
| UI / 交付设计 Agent | 演示页、交付材料、客户文档和模板包装 |
| ISSUE 管理员 Agent | Issue 编号、Open / Close、阻塞归档 |

## 4. 任务分发最小字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 任务 ID | 是 | 跨 Agent 追踪 |
| 发起方 | 是 | 项目总控制人或项目总负责人 |
| 执行 Agent | 是 | 指定唯一主责 |
| 任务目标 | 是 | 本轮完成什么 |
| 输入材料 | 是 | 文档、视频、官方资料、仓库路径或用户确认 |
| 输入边界 | 是 | 哪些可作为依据，哪些不能越权使用 |
| 输出要求 | 是 | 交付物格式、路径、证据 |
| 输出边界 | 是 | 明确不能输出什么 |
| 验收标准 | 是 | 判断完成的条件 |
| 阻塞回报格式 | 是 | 缺少信息时如何回报 |

## 5. 责任溯源规则

| 问题类型 | 首查方向 | 修复动作 |
| --- | --- | --- |
| 需求不清 | 客户场景、业务智能体能力边界、交付口径是否缺失 | 产品经理补清，业务方确认 |
| 技术不可行 | 平台限制、模型能力、插件、部署、安全是否阻塞 | 技术验证 Agent 给出替代方案 |
| 实现偏差 | 业务智能体工作流、提示词、知识库或代码是否偏离 Spec | 退回主责 Agent 最小修复 |
| 验收缺失 | 是否缺少测试、截图、日志、客户演示或部署证据 | 指派验收补齐证据 |
| 消息协议不完整 | 任务 ID、边界、产物路径、会话 ID 是否缺失 | 总负责人修订协议并重新下发 |
