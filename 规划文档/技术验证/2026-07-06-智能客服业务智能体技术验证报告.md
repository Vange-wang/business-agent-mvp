# 智能客服业务智能体技术验证报告

## 1. 结论

首版采用 Dify 作为智能客服业务智能体的主平台。

Dify 能覆盖本阶段 MVP 所需能力：知识库问答、Chatflow / Workflow、变量、条件分支、API/WebApp 发布、对话日志和后续 API 集成。Coze / Coze Studio 可作为交互和插件设计参考，但不作为首版主平台。

## 2. Dify 可覆盖能力

| MVP 能力 | Dify 支持方式 | 结论 |
| --- | --- | --- |
| FAQ / RAG 问答 | Knowledge + 检索节点 + LLM 回复 | 可行 |
| 线索收集 | Chatflow 变量、表单式追问、结构化输出 | 可行 |
| 转人工兜底 | 条件分支 + 固定话术 + 联系方式 | 可行 |
| Web 演示 | Dify WebApp / Embed | 可行 |
| API 集成 | Dify Chatflow API | 可行 |
| 日志回看 | Dify 内置日志 / 应用观测 | 可行 |
| 后续自动化 | HTTP 节点 / API / Webhook | 后续增强 |

## 3. 首版技术栈

- Dify Self-hosted 或 Dify Cloud 试点。
- Docker / Docker Compose。
- PostgreSQL。
- Redis。
- 向量库：首版按 Dify 默认方案，后续按部署规模调整。
- 模型供应商：OpenAI 兼容接口、DeepSeek、Qwen、Claude 等按客户预算选择。
- 前端嵌入：Dify WebApp / iframe / 后续 Next.js 嵌入页。

## 5. Dify 搭建建议

应用类型：

- 首版建议 Chatflow。
- 若后续需要复杂工具调用和 CRM 同步，再升级为 Workflow + API 集成。

核心节点：

1. Start：接收用户输入。
2. 意图识别：判断 FAQ、价格、预约、售后、投诉、购买意向、人工请求。
3. Knowledge Retrieval：检索商家知识库。
4. LLM Answer：生成保守回答。
5. Lead Capture：当用户表达购买/预约意向时追问线索字段。
6. Human Handoff：高风险或无法回答时转人工。
7. Answer：输出最终回复。

## 6. Coze 不作为首版主平台的原因

- Coze Studio 可学习插件、工作流和多 Agent 体验。
- 但首版商业交付更看重自托管、知识库、日志、API、可复制部署和安全收口。
- Coze Studio 公开部署需要额外关注注册、代码节点、SSRF、权限等风险。
- 因此首版主线使用 Dify，Coze 作为备选验证。

## 7. 安全风险

必须避免：

- 提交真实 API Key。
- 导入真实客户隐私数据到测试仓库。
- 让浏览器直接暴露 Dify API Key。
- 未做权限收口就公开管理后台。
- 让模型生成最终报价、合同、退款承诺。
- 模型调用无成本限制。

## 8. 成本风险

成本来自：

- 模型输入输出 token。
- 知识库召回和重排。
- Dify 部署服务器。
- 后续自动化 Webhook / CRM 调用。

建议首版套餐限制：

- 每月会话数。
- 每月消息数。
- 知识库文档数量。
- 超出后升级套餐或人工评估。

## 9. 技术验收标准

- 能用样例知识库回答至少 10 个客服问题。
- 能触发线索收集。
- 能触发转人工兜底。
- 能提供 WebApp 或本地原型演示入口。
- 能说明生产部署所需环境变量和安全边界。
