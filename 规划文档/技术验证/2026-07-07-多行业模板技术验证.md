# 多行业客服智能体模板矩阵技术验证

日期：2026-07-07

任务 ID：`CS-AGENT-TEMPLATE-MATRIX-2026-07-07-TECH-VERIFY`

负责人：技术验证 Agent

适用阶段：多行业客服智能体模板矩阵 + 标准商业化交付包

## 1. 结论

本阶段可以沿用 Dify 作为主交付平台，把已经完成的智能客服 MVP 和第二客户复制试点沉淀为“可复制模板 + 行业知识包 + 验收清单”。

技术路径不需要先做复杂多租户、计费后台或私有模型微调。马上交付优先级如下：

1. 复制一套 Dify Chatflow 模板。
2. 替换行业知识库和行业提示词。
3. 按固定召回测试集验收 RAG。
4. 按客户交付方式选择 WebApp、embed 或 API。
5. 保留上一个稳定模板作为回滚版本。

## 2. 模板矩阵范围

本阶段首批只沉淀 3 类客服模板，均复用同一条技术骨架，并作为本阶段验收范围：

| 行业模板 | 知识库重点 | 线索字段 | 高风险转人工场景 |
| --- | --- | --- | --- |
| 教育培训客服 | 课程、师资、价格区间、试听、退费流程 | 学生年级、科目、城市、试听时间、联系方式 | 退费争议、升学承诺、教师资质争议 |
| 家政清洁客服 | 服务项目、服务范围、预约流程、价格区间、售后规则 | 服务类型、服务面积、所在区域、预约时间、联系方式 | 到家安全、退款投诉、最终报价、服务纠纷 |
| 本地门店客服 | 门店地址、营业时间、服务/商品项目、预约、会员活动 | 门店/区域、需求类型、到店时间、联系方式 | 投诉、退款、活动解释、超范围承诺 |

共同能力边界：

- 可以回答资料内的 FAQ、流程、项目介绍和预约前置问题。
- 可以收集销售线索并提示人工跟进。
- 可以对不确定、高风险、资料缺失问题转人工。
- 不直接承诺最终价格、合同结果、退款结果、医疗诊断、升学结果或法律结论。

后续扩展，不进入本阶段验收：

- 装修/家居咨询：可复用模板骨架，但合同、最终报价、工期和质量纠纷风险更高，需单独补充行业风险边界后再纳入。
- 企业服务/SaaS：可作为后续 B2B 客服/售前模板，但涉及实施范围、数据安全和定制报价，不进入当前首批交付验收。
- 医美/健康咨询前置客服：仅作为风险提示方向保留，涉及诊断、疗效、用药和医疗合规边界，本阶段不做模板验收。

## 3. Dify 复制模板路径

每交付一个新行业或新客户，按以下路径复制，不从零搭建：

1. 在 Dify 中保留一个基准应用：`客服模板-基准版-v1`。
2. 复制应用为：`客服模板-{行业}-{客户简称}-v1`。
3. 复制或新建知识库：`KB-{行业}-{客户简称}-v1`。
4. 在复制后的 Chatflow 中替换 Knowledge Retrieval 节点绑定的知识库。
5. 替换 System Prompt 中的行业词、禁答边界、线索字段和转人工话术。
6. 保留节点结构不变，除非客户明确需要新增工具调用。
7. 发布前记录版本号：应用名、知识库名、模型名、发布时间、负责人。

推荐基准 Chatflow 节点：

1. Start：接收用户问题、来源渠道、客户行业变量。
2. 意图识别：FAQ、价格、预约、售后、投诉、购买意向、人工请求、超范围问题。
3. 条件分支：普通问答、线索收集、转人工、高风险拒答。
4. Knowledge Retrieval：绑定当前客户知识库。
5. LLM Answer：基于召回内容生成回答，禁止编造资料外承诺。
6. Lead Capture：逐步追问必要线索字段。
7. Human Handoff：输出人工承接口径和客户配置的联系方式。
8. Answer：输出最终回复。

复制完成后必须做一次“模板污染检查”：

- 应用名、开场白、行业名、客户名不能残留上一个客户信息。
- 转人工联系方式不能残留旧客户电话、微信、邮箱或门店地址。
- 知识库绑定不能指向旧客户知识库。
- 测试问题中不能回答出旧客户产品、价格、活动或政策。

## 4. 知识库导入规范

每个客户知识库按 4 个文件夹或 4 组文档组织：

| 知识组 | 内容 | 导入要求 |
| --- | --- | --- |
| 01-FAQ | 高频问答、价格区间、预约方式、售后问题 | 一问一答，问题口语化，答案不超过 300 字 |
| 02-产品服务 | 课程/服务/套餐/产品说明、适用人群、限制条件 | 按产品或服务拆分，避免整本 PDF 一次性导入 |
| 03-业务流程 | 咨询、预约、付款、交付、售后、转人工流程 | 用步骤列表表达，明确下一步动作 |
| 04-风险边界 | 禁止承诺、必须转人工、敏感问题口径 | 单独成文，方便召回和提示词引用 |

文档格式建议：

- 优先使用 Markdown、TXT、DOCX 转 Markdown 后导入。
- PDF 只用于版式简单、文本可复制的资料；扫描件先 OCR。
- 单篇文档建议控制在 800 到 2000 字。
- 标题必须包含行业、主题和适用范围，例如：`教育培训-退费流程-门店通用`。
- 每条政策必须写明适用条件和例外情况。
- 不导入真实客户身份证、手机号清单、订单明细、聊天记录原文或 API Key。

FAQ 标准格式：

```text
Q：你们有哪些课程？
A：我们提供小学、初中、高中阶段的数学、英语、语文等课程。具体班型和时间以门店老师确认为准。

Q：可以先试听吗？
A：可以。用户留下年级、科目、城市和联系方式后，由人工老师安排试听时间。
```

风险边界标准格式：

```text
触发问题：用户要求保证升学、保证疗效、立即退款、合同法律解释、最终报价。
处理方式：不做最终承诺，说明需要人工确认，并收集用户联系方式与问题摘要。
推荐话术：这个问题需要由人工顾问结合你的具体情况确认，我可以先帮你记录需求并安排跟进。
```

## 5. 模型选择建议

模型选择按“客户预算 + 响应质量 + 部署地区 + 合规要求”决定，不在模板层写死。

| 场景 | 推荐策略 | 验收重点 |
| --- | --- | --- |
| 标准商用试点 | DeepSeek / Qwen / OpenAI 兼容接口择一主用 | 成本、中文客服表达、稳定性 |
| 高质量演示 | 使用回答质量更稳定的主模型，保留备用模型 | 复杂追问、边界拒答、语气 |
| 数据敏感客户 | 优先自托管 Dify + 可控模型供应商；必要时评估本地模型 | 数据边界、日志留存、成本 |
| 低成本客户 | 低价主模型 + 更严格知识库召回 + 短回答 | 不胡编、响应时间、成本上限 |

建议配置：

- 主模型：用于正式回答和线索追问。
- 备用模型：主模型不可用时切换。
- 温度：客服问答建议 0.2 到 0.5。
- 最大输出长度：普通客服 500 到 800 tokens，避免长篇泛答。
- 禁止让模型自行编造价格、活动、政策、合同、疗效、退款结论。

模型切换验收：

1. 同一套 20 个问题分别跑主模型和备用模型。
2. 对比是否都能引用知识库范围内的信息。
3. 对比是否都能在高风险问题上转人工。
4. 记录主模型失败时的手工切换步骤。

## 6. RAG / 召回检查

RAG 验收不是“文档上传成功”，必须用固定测试集检查。

每个行业模板至少准备 30 条测试问题：

- 10 条标准 FAQ：资料中有明确答案。
- 5 条同义改写：用户用口语、错别字、简称提问。
- 5 条流程追问：预约、售后、付款、交付下一步。
- 5 条线索意向：表达想报名、想预约、想报价、想试用。
- 5 条风险问题：退款承诺、最终报价、医疗/法律/升学承诺、投诉纠纷。

通过标准：

- 标准 FAQ 命中率：至少 8/10 能答到正确知识点。
- 同义改写命中率：至少 4/5 不跑题。
- 流程追问：至少 4/5 能给出下一步动作。
- 线索意向：至少 4/5 能进入线索收集。
- 风险问题：5/5 必须拒绝最终承诺并转人工。

人工检查表：

| 检查项 | 通过标准 |
| --- | --- |
| 召回内容相关 | Dify 检索结果来自当前客户知识库，且主题匹配 |
| 回答不编造 | 没有资料外价格、优惠、承诺、门店信息 |
| 追问自然 | 缺少线索字段时逐步追问，不一次性压迫用户 |
| 边界稳定 | 高风险问题不被用户诱导做最终承诺 |
| 旧客户隔离 | 不出现其他客户名称、地址、联系方式、产品 |

若召回失败，按顺序修复：

1. 检查用户问题是否确实有资料答案。
2. 拆小过长文档，补充标题和 FAQ 问法。
3. 增加同义问法和行业常用简称。
4. 调整检索参数或启用重排。
5. 仍失败则在 Chatflow 中增加意图分支或固定兜底话术。

## 7. 三种接入路径

### 7.1 WebApp 路径

适用：

- 客户先要可访问演示链接。
- 不要求嵌入官网。
- 首次试点、销售演示、内部验收。

交付步骤：

1. Dify 发布 WebApp。
2. 配置应用名称、开场白、建议问题。
3. 用客户 Logo 和业务名称做基础包装。
4. 提供访问链接和验收测试问题。

验收办法：

- 浏览器打开 WebApp。
- 连续问 10 个标准问题、5 个线索问题、5 个风险问题。
- 检查会话日志是否可回看。
- 检查移动端浏览器是否可正常聊天。

### 7.2 Embed 路径

适用：

- 客户已有官网、落地页或帮助中心。
- 需要右下角悬浮客服入口。
- 交付周期短，不做深度前端定制。

交付步骤：

1. 从 Dify 获取嵌入脚本或 iframe 配置。
2. 在客户测试页面插入 embed。
3. 设置入口文案、位置、主题色。
4. 确认手机端不遮挡关键按钮。

验收办法：

- 打开客户测试页，确认客服入口出现。
- 桌面端和手机端各完成 5 轮对话。
- 刷新页面后入口仍可用。
- 检查浏览器控制台没有明显脚本错误。
- 确认页面没有暴露 Dify API Key。

### 7.3 API 路径

适用：

- 客户需要自定义前端 UI。
- 需要对接 CRM、企业微信、飞书、表单系统或自有后台。
- 需要隐藏 Dify API Key。

交付步骤：

1. 后端保存 `DIFY_API_BASE_URL` 和 `DIFY_API_KEY`，不要放到浏览器。
2. 前端请求自有后端接口。
3. 后端调用 Dify Chat API。
4. 线索收集结果通过 Webhook 或 CRM API 写入客户系统。
5. 对超时、模型错误、Dify 不可用设置固定兜底回复。

建议环境变量占位：

```text
DIFY_API_BASE_URL=
DIFY_API_KEY=
DIFY_APP_ID=
DIFY_WEBAPP_URL=
LEAD_WEBHOOK_URL=
LEAD_WEBHOOK_SECRET=
REQUEST_TIMEOUT_MS=30000
ENABLE_LEAD_CAPTURE=true
```

验收办法：

- 后端接口不返回或打印真实 API Key。
- 普通问答能收到 Dify 回复。
- 线索收集能生成结构化字段。
- Dify 超时或 5xx 时返回“人工跟进”兜底话术。
- CRM/Webhook 失败时不丢失用户原始问题和联系方式。

## 8. 验收测试命令或手工验收办法

如果本阶段只交付 Dify 配置和客户页面，优先用手工验收表；如果交付了 API 代理或嵌入页代码，再补命令验收。

### 8.1 Dify 手工验收

1. 登录 Dify 管理端。
2. 打开目标应用的调试窗口。
3. 导入并逐条执行行业测试集。
4. 查看每条回答的检索来源。
5. 导出或截图保留以下证据：
   - 应用名称和版本。
   - 绑定知识库名称。
   - 模型供应商和模型名。
   - 20 条核心测试问答结果。
   - 5 条风险问题转人工结果。

验收结论格式：

```text
应用：客服模板-教育培训-客户A-v1
知识库：KB-教育培训-客户A-v1
模型：主模型 xxx，备用模型 xxx
FAQ：8/10 通过
同义改写：4/5 通过
流程追问：5/5 通过
线索收集：4/5 通过
风险转人工：5/5 通过
是否允许交付：是 / 否
```

### 8.2 Embed 手工验收

1. 用桌面浏览器打开客户测试页。
2. 用手机浏览器或移动端模拟器打开客户测试页。
3. 检查入口位置、弹窗大小、滚动、输入框、关闭按钮。
4. 连续问 20 条测试问题。
5. 检查控制台错误和网络失败。

最低通过标准：

- 桌面端和手机端都能正常打开。
- 不遮挡客户页面核心 CTA。
- 不暴露 API Key。
- 对话日志在 Dify 后台可查。

### 8.3 API 命令验收

若项目提供本地 API 代理，可用以下 PowerShell 模板测试。真实地址和字段按代码开发员提供的接口调整。

```powershell
$body = @{
  user = "acceptance-test-user"
  query = "你们可以预约试听吗？"
  conversation_id = ""
} | ConvertTo-Json

Invoke-RestMethod `
  -Method Post `
  -Uri "http://localhost:3000/api/chat" `
  -ContentType "application/json" `
  -Body $body
```

API 验收必须覆盖：

- 资料内 FAQ。
- 资料外问题。
- 购买/预约意向。
- 转人工请求。
- Dify 超时或错误兜底。

如仓库存在前端或 API 工程，再由代码开发员执行对应项目命令，例如：

```powershell
npm run typecheck
npm run lint
npm test
npm run build
```

当前技术验证文件不要求运行这些命令；只有实际代码变更进入交付包时才执行。

## 9. 风险与回滚方案

| 风险 | 触发信号 | 处理方案 | 回滚方案 |
| --- | --- | --- | --- |
| 知识库污染 | 回答出现旧客户信息 | 立即下线当前发布版本，检查知识库绑定和提示词 | 恢复上一版应用和上一版知识库 |
| RAG 召回差 | FAQ 命中率低于 8/10 | 拆分文档、补 FAQ、补同义词、调整检索 | 回退到人工 FAQ 固定话术版本 |
| 模型编造承诺 | 输出最终价格、疗效、退款、合同承诺 | 降低温度，强化风险边界，增加转人工分支 | 切换到保守提示词版本 |
| 模型不可用 | 大量超时、5xx、余额不足 | 切换备用模型，降低输出长度 | 启用固定兜底回复和人工承接 |
| Embed 影响客户页面 | 遮挡按钮、移动端异常、脚本错误 | 调整入口位置和加载方式 | 移除 embed，临时改用 WebApp 链接 |
| API Key 暴露 | 浏览器源码或网络请求看到 Key | 立即撤销 Key，改为后端代理 | 关闭 API 接入，临时使用 WebApp |
| 线索丢失 | Webhook/CRM 写入失败 | 本地记录失败请求，提示人工补录 | 临时关闭自动写入，Dify 日志人工导出 |
| 客户资料不完整 | 大量问题走兜底 | 标记缺失资料清单，让客户补齐 | 降级为 FAQ 演示版，不承诺完整客服 |

发布前必须保留：

- 上一版 Dify 应用副本。
- 上一版知识库副本或导出资料。
- 当前版本测试集结果。
- 模型与环境变量占位清单。
- 客户页面 embed/API 接入位置记录。

紧急回滚顺序：

1. 停止使用当前应用发布入口。
2. 切回上一版 WebApp/embed/API 目标应用。
3. 撤销可能泄露的 API Key。
4. 通知交付负责人当前回滚原因和影响范围。
5. 在 Issue 表登记真实问题，修复后再重新验收。

## 10. 标准交付包中的技术文件

每个客户交付包至少包含：

1. Dify 应用名称、版本、发布时间。
2. 知识库文件清单和资料来源说明。
3. 模型配置说明：主模型、备用模型、温度、输出长度。
4. 接入方式说明：WebApp / embed / API。
5. 验收测试集和通过结果。
6. 风险边界和转人工话术。
7. 回滚步骤和联系人。
8. 不包含内容：复杂多租户、自动计费、客户后台、私有模型微调、大规模插件市场。

## 11. 当前阻塞

文档层无阻塞，可以进入多行业模板矩阵和标准商业化交付包制作。

真实客户上线前仍需逐单确认：

- 客户是否允许使用 Dify Cloud，或必须自托管。
- 客户提供的知识资料是否完整、可导入、无敏感隐私。
- 模型供应商、预算和调用量上限。
- WebApp、embed、API 三种接入路径中的实际选择。
- 线索进入哪里：人工查看 Dify 日志、Webhook、CRM、企业微信或表格。
