## 1. 结论

本项目主线选择 **Dify**，Coze Studio / Coze 作为学习参考和备选验证。

术语纠偏：本报告中的“Agent”需要区分两类对象。内部协作角色 Agent 是 Codex 项目分工线程；可售卖业务智能体 Agent 才是本项目真正要卖给客户的产品。业务智能体应面向真实客户、真实商家和真实业务流程，例如客服 Agent、信息回复 Agent、销售线索收集 Agent、私域运营 Agent 等，用于接入或适配其他软件，接管商家重复性动作和重复性工作。Dify / Coze 只是实现和验证这些业务智能体的工具底座。

选择 Dify 的核心原因：

- 更适合自托管和客户私有化交付。
- 工作流、业务智能体、RAG、知识库、API、日志观测和运营迭代链路更贴近“卖给客户并可复制交付”的目标。
- Docker Compose 部署路径清晰，官方文档明确列出核心服务和依赖组件。
- 可以先用 Dify 快速做客户 MVP，再在必要处补自定义插件、API 集成和前端嵌入页。

Coze Studio 的定位：

- 适合作为学习对象，参考其 Agent 创建、插件、知识库、变量、工作流、多 Agent 等交互方式。
- 适合作为备选技术验证，但不作为第一主线。
- Coze Studio 虽然已开源且使用 Apache 2.0，但公开网络部署有注册、代码节点、SSRF、权限等安全风险需要额外收口。

## 2. 对 `家教对接website` 运作逻辑的继承与调整

继承：

- 多内部协作角色 Agent 分工：总负责人、产品经理、代码开发员、ISSUE 管理员、UI 设计等角色边界继续保留。
- 任务分发协议：任务 ID、输入边界、输出边界、产出路径、验收标准、阻塞回报格式必须齐全。
- 阶段闭环：规划、实现、审查、复核、Issue、最终报告必须闭环。
- 权限边界：只读线程只做分析和回报，写入和验证集中到完全访问前台线程。

调整：

- 新增 `技术验证 Agent`：负责平台、模型、部署、安全和成本验证。
- 新增 `学习研究 Agent`：负责视频教程和竞品资料学习，不让开发线程被教程学习拖慢。
- 新增 `Agent 搭建师 Agent`：专门负责 Dify 工作流、知识库、提示词、工具和模板沉淀。
- UI 角色从“网站 UI 设计”调整为“客户演示与交付设计”，重点是可售卖包装、交付材料和演示可信度。
- ISSUE 分类新增“技术不可行”和“平台限制”，避免把平台能力边界误判为实现偏差。

## 4. 官方资料依据

- Dify GitHub：https://github.com/langgenius/dify
  - 官方介绍包含 Workflow、模型支持、Prompt IDE、RAG Pipeline、Agent capabilities、LLMOps。
- Dify Docker Compose 部署文档：https://docs.dify.ai/en/self-host/deploy/quick-start/docker-compose
  - 最低硬件要求：CPU >= 2 Core，RAM >= 4 GiB。
  - Docker Compose 版本要求：2.24.0+。
  - 启动核心服务包括 api、worker、worker_beat、web、plugin_daemon，以及 postgres、redis、nginx、ssrf_proxy、sandbox、weaviate 等依赖。
- Coze Studio GitHub：https://github.com/coze-dev/coze-studio
  - 后端 Golang，前端 React + TypeScript，微服务和 DDD 架构。
  - 支持 Agent、Workflow、Knowledge、Plugin、API 和 Chat SDK。
  - License 为 Apache 2.0。
- Coze Studio Quickstart：https://github.com/coze-dev/coze-studio/wiki/2.-Quickstart
  - Windows 启动路径为复制 `.env` 后使用 Docker Compose 启动。
  - 默认访问地址为 `http://localhost:8888/`。
  - 公开网络部署需注意注册、代码节点、SSRF 和权限风险。
- Coze Studio Model Configuration：https://github.com/coze-dev/coze-studio/wiki/3.-Model-configuration
  - 支持 OpenAI、DeepSeek、Claude、Ollama、Qwen、Gemini 等模型服务。

## 5. 技术栈建议

### 主平台

- Dify Self-hosted Community Edition。
- Docker / Docker Compose。
- PostgreSQL。
- Redis。
- Weaviate 或后续替换为更适合部署的向量库方案。
- Nginx / Caddy 反向代理。
- HTTPS 证书：生产环境使用 Let's Encrypt 或云厂商证书。

### 模型层

首批建议：

- OpenAI API 兼容模型作为主接入协议。
- DeepSeek / Qwen / OpenAI / Claude 根据客户预算和地区选择。
- 本地模型可用 Ollama 做演示或隐私场景验证，但不默认作为商业交付主模型。

### 应用集成

- Dify Web App / Embedding。
- Dify API。
- 自定义工具或插件。
- 客户网站嵌入页：Next.js + TypeScript + React。
- 自动化集成：n8n 可作为后续增强，不进入首个 MVP。

### 交付与运维

- `.env.example` 管理环境变量说明。
- Docker Compose 生产部署说明。
- 数据备份策略：PostgreSQL、向量库、上传文件、Dify 配置。
- 监控：容器健康、应用日志、模型调用成本、知识库命中率、用户反馈。

## 6. Dify 初始配置方法

本地技术验证路径：

```powershell
git clone --branch "$(curl -s https://api.github.com/repos/langgenius/dify/releases/latest | jq -r .tag_name)" https://github.com/langgenius/dify.git
cd dify/docker
cp .env.example .env
docker compose up -d
```

Windows 注意：

- 官方建议 Windows 使用 WSL2，并将源码和容器绑定数据放在 Linux 文件系统。
- 需要 Docker Desktop 和 Docker Compose 2.24.0+。
- 最低 2 Core / 4 GiB，建议开发机给 Docker 至少 2 Core / 8 GiB。

初始化顺序：

1. 启动 Dify。
2. 创建管理员账号。
3. 配置模型供应商。
4. 创建知识库。
5. 上传测试资料。
6. 创建 Chatflow / Workflow。
7. 配置提示词、变量、检索节点、LLM 节点、输出格式。
8. 发布 Web App 或 API。
9. 编写客户交付说明和测试用例。

## 8. 首个可售卖业务智能体建议

建议首个 MVP 做“企业知识库客服 + 销售线索收集业务智能体”。

原因：

- 客户理解成本低。
- 资料准备标准化。
- Dify RAG 和 Workflow 能力能直接发挥价值。
- 可复制到教育、装修、法律咨询、财税、医疗咨询前置问答、本地生活服务等行业。
- 容易包装为低价试点包、中价标准包、高价私有化包。

最小功能：

1. 客户上传 FAQ、产品资料、服务流程。
2. 业务智能体回答常见问题。
3. 遇到购买意向时收集姓名、联系方式、需求类型。
4. 不能回答时给出转人工口径。
5. 管理侧可以查看对话日志和线索。
6. 提供网页嵌入或独立访问链接。

## 10. 当前风险

- Dify License 是基于 Apache 2.0 增加额外条件，商业包装前需要专门确认不做与 Dify 直接竞争的平台服务。
- 客户数据和知识库涉及隐私，必须在合同和部署方案中明确数据归属、备份、删除和模型调用边界。
- 模型成本会影响利润率，必须在产品套餐中设置调用量、知识库大小和维护范围。
- Coze Studio 公开部署需要安全加固，不适合未验证前直接暴露公网。
- 视频教程质量参差不齐，学习研究 Agent 必须提炼可复用方法，不直接照搬教程项目。
