# 智能客服业务智能体商业化复制 v1 Spec

日期：2026-07-07

任务 ID：`CS-AGENT-COMMERCIAL-V1-2026-07-07`

负责人：项目总负责人 Agent

阶段定位：从 MVP 阶段进入可商用复制交付阶段

## 1. 阶段目标

本阶段目标不是重新验证“智能客服能不能做”，而是把已经验收的智能客服业务智能体 MVP 升级为可卖、可交付、可复制的标准客户项目包。

商业化复制 v1 必须满足：

1. 有明确售前定位和适用客户。
2. 有客户需求采集、知识库整理、Dify 搭建、网站嵌入、验收、运维的完整流程。
3. 有首个真实接入案例作为复制样板。
4. 有上线前验收标准和上线后维护标准。
5. 有不做范围和风险边界，避免把智能体包装成不受控的自动决策系统。

本阶段继续严格区分：

- 内部协作角色 Agent：用于项目协作、任务分发和验收。
- 可售卖业务智能体 Agent：面向客户交付、接入真实业务流程、接管重复客服工作的商业产品。

## 2. 产品定义

产品名称：智能客服业务智能体商业化复制 v1

主平台：Dify

主模型建议：`deepseek-v4-flash`

升级模型建议：`deepseek-v4-pro`

交付形态：

- Dify Chatflow / Knowledge / WebApp。
- 客户网站嵌入入口。
- 客户知识库 Markdown。
- 客户验收清单。
- 运维维护手册。
- 可选二阶段 API 中转方案。

客户购买的不是 Dify 账号，也不是一次提示词配置，而是一套能嵌入其业务入口、基于其资料回答客户问题、收集线索并转人工的业务客服能力。

## 3. 目标客户准入

适合首批复制交付的客户：

- 有稳定 FAQ、服务流程、价格范围和转人工方式。
- 每天存在重复咨询，人工客服有明显重复回复成本。
- 允许先用 WebApp 或 iframe 入口上线试点。
- 能提供客户确认版知识库资料。
- 接受“智能客服回答常规问题，高风险问题转人工”的边界。

暂不建议进入商业化复制 v1 的客户：

- 需要一开始就深度接入复杂 CRM、支付、订单、合同审批或内部 ERP。
- 需要智能体自动做退款、赔偿、正式报价、合同判断等高风险决策。
- 无法提供稳定业务资料，却要求智能体自行编写业务规则。
- 对数据合规、模型供应商或部署方式有严格要求，但尚未完成合规确认。

## 4. 标准交付流程

### 4.1 售前确认

使用 `交付文档/智能客服业务智能体交付包v1/01-售前一页纸.md` 说明能力和边界。

输出：

- 客户是否适合首版交付。
- 客户确认的上线入口。
- 客户确认的试点范围和报价。

### 4.2 需求采集

使用 `交付文档/智能客服业务智能体交付包v1/02-客户需求采集表.md` 采集资料。

必须确认：

- 服务内容。
- 价格范围或报价原则。
- 预约或下单流程。
- 售后、退款、投诉、合同边界。
- 转人工方式。
- 线索字段。
- 12 个以上客户真实验收问题。

### 4.3 知识库整理

使用 `交付文档/智能客服业务智能体交付包v1/03-客户知识库整理模板.md` 整理 Markdown。

规则：

- 只写客户确认过的内容。
- 冲突内容先列为缺口，不直接导入。
- 价格、退款、合同、投诉统一采用保守口径。
- 不能导入身份证、银行卡、家庭住址、内部员工私人联系方式等高敏感信息。

### 4.4 Dify 搭建

复制现有 Dify 模板材料：

- `Agent研究文档/Dify模板/智能客服业务智能体/system-prompt.md`
- `Agent研究文档/Dify模板/智能客服业务智能体/workflow-blueprint.json`
- `Agent研究文档/Dify模板/智能客服业务智能体/test-cases.md`

最小链路：

```text
Start
-> Knowledge Retrieval
-> LLM
-> Answer
```

发布前要求：

- 知识库已上传。
- 检索节点已连接。
- 系统提示词已配置。
- 主模型已选定。
- WebApp 已发布。

### 4.5 网站接入

首版优先使用 WebApp 或 iframe 接入。

当前接入样板：

- 客户网站：家教对接 website。
- 入口路径：`/customer-service`。
- 环境变量：`NEXT_PUBLIC_DIFY_CUSTOMER_SERVICE_URL`。
- Dify WebApp：必须使用 Dify 发布页提供的 `https://udify.app/chatbot/{WEBAPP_TOKEN}`。
- 生产环境：腾讯云 CloudBase。

纠偏记录：

- `https://udify.app/chat/{APP_ID}` 会触发 `App with code ... not found`，不能作为生产验收通过依据。
- Dify 后台应用 ID 和 WebApp 发布 token 必须区分。

二阶段再进入 API 中转：

- 需要自定义聊天 UI。
- 需要隐藏 Dify API Key。
- 需要线索同步到表格、CRM、飞书、企微或数据库。
- 需要统一限流、日志、错误兜底和客户隔离。

### 4.6 验收与交付

使用 `交付文档/智能客服业务智能体交付包v1/06-验收清单.md` 执行验收。

验收必须覆盖：

- 常规问答。
- 价格保守回复。
- 预约或咨询线索收集。
- 转人工。
- 未知问题兜底。
- 网站入口可访问。
- 移动端可用。
- 客户能查看对话记录或 Dify 日志。

### 4.7 运维复盘

使用 `交付文档/智能客服业务智能体交付包v1/07-运维维护手册.md` 完成培训。

上线后 7 天内必须复盘：

- 未回答问题。
- 回答不准问题。
- 高频咨询问题。
- 转人工原因。
- 新增知识库条目。

## 6. 商业化复制 v1 不做范围

本阶段不默认交付：

- 多租户 SaaS 后台。
- 客户登录权限系统。
- 自动计费和支付系统。
- 私有模型微调。
- 复杂 CRM 双向同步。
- 自动合同、退款、赔偿、法律判断。
- 未经确认的数据合规承诺。
- 生产密钥、客户隐私数据或真实数据库备份入仓。

## 7. 阶段验收标准

满足以下条件可判定商业化复制 v1 阶段通过：

1. Dify 真实应用已发布并可访问，且 WebApp 地址使用 `https://udify.app/chatbot/{WEBAPP_TOKEN}`。
2. 至少一个真实网站入口完成接入并通过生产环境验证，页面不得出现 `App with code ... not found`。
3. 交付包覆盖售前、需求、知识库、SOP、报价、验收、运维、API 二阶段。
4. 有首个接入复盘和复制模板。
5. 有明确 Issue 状态，当前不存在阻塞 Open Issue。
6. 有阶段验收报告，结论明确为通过或不通过。

## 8. 下一阶段建议

商业化复制 v1 通过后，下一阶段进入“二客户复制试点”：

1. 选择第二个真实或模拟客户行业。
2. 按同一交付包复制一份客户专属目录。
3. 替换知识库、提示词边界和验收问题。
4. 在 Dify 中复制应用并重新发布。
5. 对比首个案例的交付耗时、问题数量和知识库缺口。
6. 若两个客户均能闭环，再考虑 API 中转、线索同步和客户后台。
