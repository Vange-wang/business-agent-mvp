# 二客户复制试点 Spec

日期：2026-07-07

任务 ID：`CS-AGENT-SECOND-CUSTOMER-PILOT-2026-07-07`

负责人：项目总负责人 Agent

阶段定位：商业化复制 v1 后的第二客户复制验证

## 1. 阶段目标

本阶段目标是验证智能客服业务智能体不只适用于首个家教网站接入案例，也能按同一套交付包复制到第二个不同类型客户。

试点选择：本地家政清洁服务商家。

样板客户名称：清安到家家政清洁。

本阶段验证以下复制能力：

1. 同一套需求采集、知识库整理、Dify 配置、验收和运维流程可复用。
2. 同一套客服核心能力能处理不同业务行业。
3. 第二客户知识库不会复用首个客户的业务内容。
4. 第二客户测试能覆盖服务范围、价格、预约、售后、转人工和线索收集。
5. 当前没有阻塞 Issue 后，才允许进入多行业模板沉淀阶段。

## 2. 试点客户场景

清安到家家政清洁是一家本地生活服务商家，主要提供：

- 日常保洁。
- 深度清洁。
- 开荒保洁。
- 厨房油污清洁。
- 卫生间深度清洁。
- 玻璃清洁。

该客户有大量重复咨询：

- 服务项目有哪些。
- 开荒保洁多少钱。
- 是否能今天上门。
- 服务哪些区域。
- 清洁不满意怎么办。
- 如何预约。

## 3. 复制范围

| 资产 | 复制方式 |
| --- | --- |
| 售前和需求采集流程 | 复用 `智能客服业务智能体交付包v1` |
| 知识库结构 | 替换为家政清洁客户专属知识库 |
| Dify Chatflow | 复用 `workflow-blueprint.json`，替换 Knowledge 数据集和应用名称 |
| 系统提示词 | 复用通用客服边界，补充家政行业不承接范围 |
| 验收清单 | 使用客户真实问题和自动化测试共同验收 |
| 运维说明 | 复用交付包运维手册，强调服务项目、价格和售后变更必须更新知识库 |

## 4. 不做范围

本阶段不做：

- 给清安到家上线真实生产网站。
- 接入真实订单系统、支付系统或排班系统。
- 自动计算最终报价。
- 自动处理退款、赔偿、投诉。
- 处理真实客户隐私数据。
- 建设多租户后台。

原因：二客户复制试点的目标是验证交付方法和知识库复制能力，不是扩展为完整 SaaS。

## 5. 验收标准

二客户复制试点必须满足：

1. 有第二客户独立交付目录。
2. 有第二客户专属知识库。
3. 有 Dify 复制配置说明。
4. 自动化测试覆盖第二客户知识库加载。
5. 自动化测试覆盖服务范围问答。
6. 自动化测试覆盖价格保守回复。
7. 自动化测试覆盖预约线索收集。
8. 自动化测试覆盖投诉、退款、赔偿转人工。
9. 原有智能客服业务智能体测试不能回退。
10. Issue 总表无阻塞 Open Issue。

## 6. 阶段通过条件

满足以下条件可判定本阶段通过：

- `node --test` 对本地智能客服原型全部通过。
- 第二客户交付包可独立阅读并复制执行。
- 阶段验收报告结论为通过。
- Issue 总表无阻塞 Open Issue。

通过后进入下一阶段：多行业模板沉淀。
