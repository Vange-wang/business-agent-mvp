# 智能客服业务智能体 MVP Spec

日期：2026-07-06

任务 ID：`CS-AGENT-MVP-2026-07-06-001`

负责人：产品经理 Agent

落盘执行：项目总负责人前台线程

## 1. 背景和目标

本阶段要交付第一个可售卖业务智能体：智能客服业务智能体。

它不是 Codex 协作线程，也不是内部角色名，而是面向真实客户和真实商家的商业化业务智能体产品。首版目标是接管商家重复客服工作，包括常见问题回复、业务资料问答、基础咨询分流、销售线索收集、无法回答时转人工和对话记录留存。

首版以 Dify 作为实现底座，Dify 是工具和运行平台，客户购买的是可接入其业务流程的智能客服能力。

## 2. 目标客户与真实业务场景

首批目标客户：

- 本地生活服务商家，例如家政、装修、摄影、维修、培训咨询。
- 有大量重复咨询的中小企业。
- 有固定 FAQ、产品介绍、服务流程和报价规则的商家。
- 暂时没有能力搭建完整客服系统，但希望减少人工重复回复的团队。

真实场景：

- 客户在官网、落地页、公众号菜单、私域链接或广告页中进入客服。
- 智能客服根据商家知识库回答服务内容、价格范围、预约方式、售后规则等问题。
- 当客户出现购买意向时，智能客服收集线索。
- 当问题超出知识库或涉及人工判断时，智能客服转人工。

## 3. 用户角色

| 角色 | 说明 |
| --- | --- |
| 访客 / 潜在客户 | 向智能客服咨询服务、价格、流程、售后、预约方式 |
| 商家客服 / 销售 | 接收智能客服收集的线索，处理复杂问题和人工跟进 |
| 商家运营者 | 准备知识库资料，查看对话和线索，优化话术 |
| 项目交付方 | 帮商家搭建、测试、部署和维护智能客服业务智能体 |

## 4. 核心业务流程

1. 商家提供 FAQ、产品/服务说明、价格规则、预约流程、售后规则、联系方式和转人工规则。
2. 项目交付方整理资料并导入 Dify Knowledge。
3. 搭建 Dify Chatflow / Workflow。
4. 访客提问。
5. 智能客服检索知识库并生成回答。
6. 如果访客表达咨询/预约/购买意向，智能客服收集线索字段。
7. 如果问题无法回答、涉及承诺、投诉、退款、合同、隐私或高风险内容，智能客服给出转人工口径。
8. 商家在 Dify 日志或后续线索表中查看对话和线索。

## 5. MVP 功能范围

### 5.1 FAQ 与业务资料问答

- 支持回答商家服务内容、服务流程、营业时间、服务区域、价格范围、预约方式、售后规则。
- 回复必须基于知识库，不编造没有依据的信息。
- 对价格、优惠、合同、退款等敏感内容使用保守口径。

### 5.2 知识库检索

- 支持导入 Markdown、PDF、TXT、网页复制内容等资料。
- 首版至少准备一份样例知识库，用于演示。
- 知识库内容必须按真实商家可维护的结构组织。

### 5.3 销售线索收集

首版线索字段：

- 姓名或称呼。
- 联系方式。
- 需求类型。
- 所在城市或服务区域。
- 预算或期望价格范围，可选。
- 期望联系时间，可选。
- 备注。

原则：

- 不强制收集身份证、银行卡、家庭住址等高敏感信息。
- 未经客户同意，不承诺自动发送给第三方系统。

### 5.4 转人工和兜底

触发条件：

- 知识库没有答案。
- 用户要求投诉、退款、合同、开票、正式报价。
- 用户输入明显涉及隐私、违法、高风险承诺。
- 用户明确要求人工服务。

兜底话术：

> 这个问题需要人工进一步确认。我可以先帮你记录需求，请留下称呼、联系方式和方便联系的时间，工作人员会尽快跟进。

### 5.5 对话记录和运营回看

- 首版使用 Dify 内置日志和会话记录。
- 后续可扩展到 CRM、表格、飞书、企微或数据库。

### 5.6 演示入口

- 首版优先使用 Dify WebApp 或嵌入方式。
- 同步提供一个本地 HTML 原型，用于无模型 Key 情况下演示业务流程。

## 6. 不做范围

首版不做：

- 多租户客户后台。
- 支付、套餐、自动计费。
- 客户登录权限系统。
- 真实 CRM 双向同步。
- 微信、企微、抖音、小红书等平台深度接入。
- 私有模型微调。
- 自动生成正式报价、合同、退款决定。
- 处理真实客户隐私数据或真实生产密钥。

## 7. 知识库资料要求

商家至少提供：

- 商家简介。
- 服务/产品清单。
- 常见问题和标准答案。
- 价格或价格范围说明。
- 服务区域和服务时间。
- 预约流程。
- 售后/退款/投诉处理原则。
- 转人工联系方式和工作时间。

## 8. 智能客服回复策略

- 先理解用户意图，再检索知识库。
- 回答应简短、明确、自然，避免一次性输出过长。
- 能回答时直接回答，并给下一步建议。
- 不确定时说明需要人工确认，不编造。
- 对购买意向主动引导留资。
- 对高风险问题转人工。

## 9. 数据与隐私边界

- 所有真实密钥不得写入仓库。
- 演示数据必须使用样例数据。
- 线索字段只收集完成咨询所需的最少信息。
- 客户数据归属、存储、删除、备份和模型调用边界必须在正式合同中另行确认。

## 10. 演示验收标准

必须通过以下演示：

1. 能回答服务内容类问题。
2. 能回答价格范围类问题，并避免承诺最终报价。
3. 能回答预约流程类问题。
4. 对知识库没有的问题能兜底转人工。
5. 能识别购买/预约意向并收集线索。
6. 能展示线索字段。
7. 能说明首版采用 Dify WebApp / Embedding 或本地原型演示。
8. 能提供客户交付说明和后续部署清单。

## 11. 后续迭代建议

- 接入 CRM / 表格 / 飞书 / 企微 Webhook。
- 增加客户行业模板。
- 增加对话质量评分和知识库缺口分析。
- 增加多渠道接入。
- 增加客户侧配置界面。
- 增加套餐、用量和成本监控。
