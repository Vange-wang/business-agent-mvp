# 多行业客服智能体模板矩阵 + 标准商业化交付包 Spec

日期：2026-07-07

任务 ID：`CS-AGENT-MULTI-INDUSTRY-TEMPLATE-MATRIX-2026-07-07`

负责角色：产品经理 Agent

阶段定位：智能客服 MVP 与第二客户复制试点之后的产品化沉淀阶段

## 1. 阶段目标

本阶段目标不是重新证明“智能客服能不能做”，而是把已经验证过的智能客服 MVP 和第二客户复制经验，沉淀成可面向更多客户复用的“多行业客服智能体模板矩阵 + 标准商业化交付包”。

本阶段只先做 3 个行业模板：

1. 教育培训客服模板。
2. 家政清洁客服模板。
3. 本地门店客服模板。

阶段完成后，应形成一套可复用的商业化交付资产，使后续客户项目不再从零开始写需求、整理知识库、配置 Dify 应用、设计验收问题和交付说明。

## 2. 目标客户

优先服务以下客户类型：

- 有明确服务项目、价格区间、预约流程和售后规则的中小商家。
- 每天存在大量重复咨询，人工客服需要反复回答相似问题。
- 可以接受先用 Dify WebApp、iframe 或轻量嵌入方式上线试点。
- 能提供已确认的 FAQ、服务说明、价格规则、门店 / 服务区域、联系方式和转人工方式。
- 接受“智能客服回答常规问题，复杂、高风险、争议性问题转人工”的边界。

本阶段不优先服务以下客户：

- 一开始就要求完整多租户 SaaS 后台、支付、订单、排班、CRM 深度双向同步的客户。
- 要求智能体自动承诺退款、赔偿、合同结论、医疗效果、法律判断或最终报价的客户。
- 业务资料不稳定、无法确认知识库内容，却要求智能体自行生成经营规则的客户。
- 对私有化、合规审计、数据跨境、模型供应商等有严格要求但尚未完成确认的客户。

## 3. 模板矩阵范围

### 3.1 教育培训客服模板

适用场景：

- 家教平台、课外辅导机构、本地教育咨询服务、培训课程咨询。

核心问答范围：

- 可辅导科目、年级、老师类型和授课方式。
- 价格区间、试听课、服务流程和匹配规则。
- 家长如何提交需求、如何预约沟通、如何转人工。
- 不承诺提分结果、不承诺教师资质外的未经确认信息。

交付资产：

- 行业知识库 Markdown 模板。
- 行业 FAQ 样例。
- 行业提示词边界。
- 行业验收问题清单。

### 3.2 家政清洁客服模板

适用场景：

- 保洁、开荒、深度清洁、上门维修、到家生活服务商家。

核心问答范围：

- 服务项目、服务区域、预约方式和服务时长。
- 价格区间、上门前准备、售后反馈和投诉转人工。
- 用户如何留下姓名、电话、地址区域、期望服务时间等线索。
- 不自动承诺最终报价、退款、赔偿和服务纠纷处理结果。

交付资产：

- 行业知识库 Markdown 模板。
- 行业 FAQ 样例。
- 行业提示词边界。
- 行业验收问题清单。

### 3.3 本地门店客服模板

适用场景：

- 餐饮门店、零售门店、宠物门店、维修门店、社区服务门店等低风险本地商家。

核心问答范围：

- 门店地址、营业时间、服务 / 商品范围、到店方式和联系方式。
- 预约、排队、咨询、售后反馈和转人工。
- 用户如何留下姓名、电话、到店时间、需求描述等线索。
- 不自动承诺库存、最终报价、退款、赔偿和客诉处理结果。

交付资产：

- 行业知识库 Markdown 模板。
- 行业 FAQ 样例。
- 行业提示词边界。
- 行业验收问题清单。

## 4. MVP 以外的新增能力

相对于已完成的智能客服 MVP，本阶段新增能力只做“可复制产品化沉淀”，不做平台级扩张。

新增能力包括：

1. 行业模板矩阵：为 3 个行业分别沉淀知识库结构、FAQ、提示词边界和验收问题。
2. 标准客户准入判断：在售前阶段判断客户是否适合首版客服智能体交付。
3. 标准需求采集表升级：从通用客服问题扩展到行业字段，例如科目 / 服务项目 / 门店 / 服务区域 / 预约时段。
4. 标准交付包目录：每个客户项目必须能复制出独立目录，避免混用其他客户知识库。
5. 标准验收清单：每个行业至少包含常规问答、价格保守回复、预约 / 线索收集、未知问题兜底、转人工、高风险问题拒答或转人工。
6. 交付说明统一口径：客户购买的是可嵌入业务入口的客服能力，不是 Dify 账号、Codex 会话或一段孤立提示词。

美业、轻医美和强到店预约类咨询不进入首批 P0 模板。它们可作为后续扩展方向单独研究，但必须先明确效果承诺、医疗健康边界、退款客诉、资质表达和人工介入规则，不能直接套用本阶段的低风险门店模板。

## 5. 标准商业化交付包

本阶段交付包应包含：

| 资产 | 说明 |
| --- | --- |
| 售前说明 | 面向客户说明能力、边界、适用行业和不承诺事项 |
| 需求采集表 | 收集 FAQ、服务范围、价格规则、预约流程、转人工方式、线索字段 |
| 行业知识库模板 | 3 个行业各 1 份 Markdown 结构模板 |
| 行业提示词模板 | 3 个行业各 1 份系统提示词边界模板 |
| Dify 搭建说明 | 说明 Knowledge、Chatflow / Workflow、WebApp 发布和复制步骤 |
| 验收清单 | 3 个行业各 1 份验收问题清单 |
| 客户交付说明 | 说明上线入口、日常维护、知识库更新和转人工规则 |
| 运维复盘表 | 上线后记录未知问题、错误回答、高频问题和知识库补充项 |

## 6. 与 Dify / Coze 的关系

Dify 是本阶段主平台，用于承载可交付的业务智能体模板、知识库、Chatflow / Workflow、WebApp 发布和后续复制。

Coze / Coze Studio 作为备选学习和交互参考平台，可用于观察插件、工作流体验、对话设计和竞品能力，但本阶段不把 Coze 作为主交付底座。

客户购买的不是 Dify 或 Coze 本身，而是一套基于客户业务资料搭建、可嵌入真实业务入口、能处理重复咨询和线索收集的客服智能体交付结果。

## 7. 术语边界

本项目继续严格区分两类 Agent：

- 内部协作角色 Agent：项目总负责人、产品经理、技术验证、学习研究、Agent 搭建师、代码开发员、UI / 交付设计、ISSUE 管理员等 Codex 协作角色。它们负责项目管理、研发、验证、文档和交付协同，不是卖给客户的产品。
- 可售卖业务智能体 Agent：面向真实客户和真实商家交付的客服智能体产品。它基于 Dify 等平台搭建，接入客户知识库和业务流程，用于回答重复问题、收集线索、引导预约和转人工。

本 Spec 中的“多行业客服智能体模板矩阵”指可售卖业务智能体的行业交付模板，不指内部 Codex 线程或内部协作角色。

## 8. 验收标准

本阶段通过条件：

1. 已形成 3 个行业模板：教育培训、家政清洁、本地门店。
2. 每个行业模板都有知识库结构、FAQ 样例、提示词边界和验收问题清单。
3. 标准商业化交付包能支持从售前、需求采集、知识库整理、Dify 搭建、WebApp / iframe 接入、验收、运维复盘的完整流程。
4. 每个行业模板都明确高风险问题处理边界，不能自动承诺退款、赔偿、最终报价、诊断、法律结论或服务结果。
5. 任意新客户复制时，必须使用客户专属知识库和客户专属交付目录，不得混用其他客户资料。
6. Dify 应用复制说明清晰，能让 Agent 搭建师按模板复制并替换 Knowledge、应用名称、欢迎语、提示词和验收问题。
7. 客户交付说明中明确：智能客服负责常规问答、线索收集和转人工，不替代人工最终决策。
8. ISSUE 管理员确认不存在阻塞下一阶段的 Open Issue。

## 9. 明确不做事项

本阶段不做：

- 多租户 SaaS 平台。
- 客户登录后台和复杂权限系统。
- 支付、计费、订单、排班、合同或发票系统。
- 复杂 CRM / ERP 双向同步。
- 私有模型微调。
- 大规模插件市场。
- 自动退款、赔偿、仲裁、法律判断、医疗诊断或最终报价。
- 将美业、轻医美、强效果承诺的到店预约咨询作为首批 P0 模板。
- 真实客户隐私数据、生产密钥、数据库密码入库。
- 把 Dify / Coze 包装成本项目最终产品本身。
- 把内部协作 Agent 当成对外售卖的业务智能体。

## 10. 下一阶段建议

本阶段通过后，下一阶段可进入“3 行业模板实装与交付资产落地”，由对应角色分工：

- Agent 搭建师：按本 Spec 创建教育培训、家政清洁、本地门店 3 个 Dify 模板与行业提示词。
- 学习研究 Agent：补充教育培训、家政清洁、本地门店 3 个行业的真实问答样例和竞品案例；美业 / 轻医美仅作为后续扩展研究，不进入本批 P0。
- 代码开发员 Agent：只在需要嵌入页、API 中转或自动化测试时介入。
- UI / 交付设计 Agent：包装客户演示页、交付说明和模板展示材料。
- ISSUE 管理员：登记模板落地过程中的阻塞问题。

产品经理 Agent 继续负责范围控制、验收口径、客户准入标准和不做事项边界。
