# 三行业 FAQ 测试集与可视化预览 Spec

日期：2026-07-07

任务 ID：`CS-AGENT-THREE-INDUSTRY-IMPLEMENTATION-VISUAL-2026-07-07`

负责角色：产品经理 Agent

阶段定位：三行业模板矩阵之后的实装验收与销售展示准备阶段

## 1. 阶段目标

本阶段目标是把已确认的三行业智能客服模板，从“模板矩阵说明”推进到“可验收、可展示、可销售说明”的交付状态。

本阶段只聚焦三类成果：

1. 三行业 FAQ 测试集：为教育培训、家政清洁、本地门店分别整理可执行的测试问题、预期回答口径和风险边界。
2. Dify 实装验收材料：明确每个行业模板在 Dify 中需要具备的 Knowledge、Chatflow / Workflow、欢迎语、提示词边界、转人工规则和验收记录。
3. 可视化预览页：把阶段成果做成客户和内部成员都能看懂的展示页，用来说明做了什么、为什么这么做、目前做到什么程度。

本阶段不是重新定义智能客服 MVP，也不是扩张为完整 SaaS 平台。它的重点是让三行业模板能被验收、被演示、被销售解释。

## 2. 行业范围

本阶段仍然只覆盖以下 3 个低风险、高频重复咨询行业：

| 行业 | 模板目标 | 核心问题类型 |
| --- | --- | --- |
| 教育培训 | 回答课程、老师、价格区间、试听、预约和转人工问题 | 科目年级、授课方式、试听流程、价格范围、家长留资 |
| 家政清洁 | 回答服务项目、服务区域、预约、价格区间、售后和转人工问题 | 日常保洁、开荒保洁、服务时长、上门区域、售后反馈 |
| 本地门店 | 回答门店信息、营业时间、服务 / 商品、到店预约和联系方式问题 | 地址时间、服务范围、预约排队、咨询留资、售后转人工 |

本阶段明确不扩张到以下高风险或强合规行业：

- 医美、轻医美、医疗健康、诊疗咨询。
- 金融、保险、投资、贷款、理财。
- 法律咨询、合同判断、纠纷裁决。
- 强效果承诺行业，例如承诺提分、承诺收益、承诺治疗效果、承诺赔偿结果的业务。

这些行业后续如要进入模板矩阵，必须单独做风险边界、合规口径、人工介入规则和验收标准，不能直接复用本阶段三行业模板。

## 3. 三行业 FAQ 测试集要求

FAQ 测试集不是简单罗列问题，而是用于验证 Dify 模板是否真的能交付。每个行业至少包含 6 类测试问题。

| 测试类别 | 验证目标 | 预期口径 |
| --- | --- | --- |
| 常规服务问答 | 验证知识库可回答核心业务问题 | 基于知识库直接回答，不编造 |
| 价格范围问答 | 验证价格表达是否保守 | 只给区间或规则，不承诺最终报价 |
| 预约 / 留资 | 验证是否能引导用户留下必要线索 | 收集称呼、联系方式、需求类型、时间或区域 |
| 未知问题兜底 | 验证知识库缺口处理 | 明确需要人工确认，不自行生成规则 |
| 转人工触发 | 验证复杂问题是否能交给人工 | 给出转人工说明并引导记录需求 |
| 风险问题边界 | 验证退款、赔偿、合同、效果承诺等问题处理 | 不做最终决定，不承诺结果，转人工 |

### 3.1 教育培训 FAQ 测试集方向

必须覆盖：

- 能否辅导某年级、某科目。
- 是否有线上 / 线下 / 上门授课。
- 试听课如何预约。
- 价格如何计算，只能说明范围或影响因素。
- 家长如何提交孩子年级、科目、所在区域和联系方式。
- 不承诺提分结果，不虚构老师资质，不代替合同或退款判断。

### 3.2 家政清洁 FAQ 测试集方向

必须覆盖：

- 日常保洁、深度清洁、开荒保洁等服务范围。
- 是否覆盖某区域、是否能某时间上门。
- 价格只能按面积、项目、时长等规则说明，不能自动最终报价。
- 用户如何留下地址区域、服务项目、期望时间和联系方式。
- 清洁不满意、退款、赔偿、投诉必须转人工。
- 不自动承诺师傅排班、最终价格和赔偿结果。

### 3.3 本地门店 FAQ 测试集方向

必须覆盖：

- 门店地址、营业时间、联系方式。
- 服务 / 商品范围和到店流程。
- 是否需要预约、排队或提前咨询。
- 用户如何留下到店时间、需求描述和联系方式。
- 售后、退款、投诉、库存、最终报价必须保守回答或转人工。
- 不自动承诺库存、折扣、赔偿和最终处理结果。

## 4. Dify 实装验收材料要求

每个行业模板在 Dify 中至少要具备以下验收材料：

| 材料 | 要求 |
| --- | --- |
| 行业 Knowledge | 使用该行业专属样例资料，不混用其他行业或客户资料 |
| 行业提示词 | 明确角色、回答范围、风险边界、线索收集字段和转人工条件 |
| Chatflow / Workflow | 能完成意图识别、知识库回答、线索收集、兜底和转人工 |
| 欢迎语 | 面向该行业用户说明可以咨询什么，不夸大能力 |
| FAQ 测试记录 | 记录测试问题、实际回答、是否通过、失败原因和修正建议 |
| 销售说明口径 | 能向客户解释该行业模板能解决什么问题、不能解决什么问题 |

Dify 实装验收的重点不是“创建了一个应用名称”，而是模板具备可复查证据：知识库内容、提示词边界、测试问题、回答结果和失败记录都能被后续角色查看。

## 5. 可视化预览页要求

可视化预览页要服务两类人：

- 内部成员：快速知道当前阶段做了哪些资产，哪些已经完成，哪些还待实装或验收。
- 潜在客户：不用阅读全部 Spec，也能理解智能客服模板能带来什么价值和边界。

预览页必须回答三件事。

### 5.1 做了什么

页面要清楚展示：

- 已将智能客服模板推进到教育培训、家政清洁、本地门店 3 个行业。
- 每个行业都有 FAQ 测试集、Dify 实装验收项和销售说明方向。
- 当前交付重点是常规问答、价格保守回复、预约 / 留资、未知问题兜底和转人工。

### 5.2 为什么这么做

页面要说明：

- 这 3 个行业重复咨询多、资料结构清晰、首版风险相对可控，适合作为智能客服商业化复制样板。
- 先做 FAQ 测试集，是为了避免只做展示页面却无法验收真实问答质量。
- 先做 Dify 实装验收材料，是为了让 Agent 搭建师、技术验证和交付设计后续能按同一口径落地。
- 先做可视化预览，是为了让用户、客户和内部角色都能看懂项目进度，而不是只能阅读长文档。

### 5.3 目前做到什么程度

页面要用状态分层展示：

| 状态 | 含义 |
| --- | --- |
| 已定义 | Spec、范围、行业边界和验收口径已经明确 |
| 待实装 | 需要 Agent 搭建师在 Dify 中创建或复制对应模板 |
| 待测试 | 需要用 FAQ 测试集跑真实问答记录 |
| 待包装 | 需要 UI / 交付设计整理成客户可看的预览页和销售说明 |
| 待验收 | 需要产品经理、技术验证、Agent 搭建师共同确认是否通过 |

本阶段预览页不要求伪装成已上线产品。它必须如实区分“已完成定义”“待 Dify 实装”“待问答测试”“待客户演示包装”。

## 6. 角色协作边界

| 角色 | 本阶段责任 |
| --- | --- |
| 产品经理 Agent | 定义目标、行业范围、测试集结构、验收标准和不做事项 |
| Agent 搭建师 Agent | 根据 Spec 在 Dify 中实装三行业模板并记录配置 |
| 技术验证 Agent | 验证 Dify 模板、知识库检索、兜底和转人工是否按预期工作 |
| UI / 交付设计 Agent | 制作可视化预览页和客户可读销售展示材料 |
| ISSUE 管理员 Agent | 登记实装、测试、展示过程中的真实阻塞 |
| 项目总负责人 Agent | 协调阶段推进、检查角色边界和闭环状态 |

本 Spec 由产品经理 Agent 负责定义。它不是 Dify 实装记录，也不是 UI 页面源码，也不是最终客户合同。

## 7. 验收标准

本阶段通过条件：

1. 已明确三行业范围：教育培训、家政清洁、本地门店。
2. 已明确不进入医美、金融、法律、医疗健康等高风险行业。
3. 每个行业都有 FAQ 测试集方向，覆盖常规问答、价格、预约 / 留资、未知兜底、转人工和风险边界。
4. Dify 实装验收材料清单完整，包括 Knowledge、提示词、Chatflow / Workflow、欢迎语、测试记录和销售说明口径。
5. 可视化预览页目标明确，必须回答“做了什么、为什么这么做、目前做到什么程度”。
6. 预览页状态表达真实，不把待实装、待测试、待验收内容包装成已上线能力。
7. 验收标准能被 Agent 搭建师、技术验证、UI / 交付设计和 ISSUE 管理员继续执行。
8. 本阶段不引入真实客户隐私数据、真实 API Key、生产密钥或客户合同信息。

## 8. 明确不做事项

本阶段不做：

- 不新增第四个行业。
- 不进入医美、金融、法律、医疗健康等高风险行业。
- 不搭建多租户 SaaS 后台。
- 不做支付、订单、排班、合同、发票、自动计费系统。
- 不做复杂 CRM / ERP 双向同步。
- 不做私有模型微调。
- 不承诺自动退款、赔偿、仲裁、法律判断、医疗诊断、投资建议或最终报价。
- 不使用真实客户隐私数据、生产密钥、数据库密码或未授权资料。
- 不把 Dify / Coze 平台本身包装成最终售卖产品。
- 不把内部协作角色 Agent 当成对外售卖的业务智能体。
- 不把可视化预览页做成夸大宣传页；必须能区分已完成、待实装、待测试和待验收。

## 9. 当前阶段结论

本阶段的核心判断是：三行业模板已经不应继续停留在“行业清单”层面，而应进入可验收资产阶段。

优先级顺序为：

1. 先补齐三行业 FAQ 测试集。
2. 再按测试集推动 Dify 实装和回答验收。
3. 最后制作可视化预览页，用清晰状态告诉用户和潜在客户：做了什么、为什么这么做、目前做到什么程度。

只要三行业测试集、Dify 实装验收材料和可视化预览页三者闭环，项目就能从“能做智能客服”推进到“能解释、能演示、能验收、能复制销售”的状态。
